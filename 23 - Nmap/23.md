## 1. What is Nmap?

### 📘 Formal Definition

**Nmap (Network Mapper)** is a free, open-source **network scanning and security auditing tool** that discovers hosts on a network, identifies open ports and running services, detects operating systems, and can execute scripts against discovered targets — all using structured probe packets and analysis of their responses.

### 🧠 Simple Explanation

Nmap is a tool that **actively interrogates a network** to answer questions like: *"Which devices are online? Which ports are open? What services are running? What OS is this device running?"* — it's the difference between passively watching traffic (Wireshark, *Chapter 22*) and actively asking questions of the network.

### 🗺️ Real-World Analogy: A Thorough Property Inspector

> Nmap is like a **building inspector going door to door** in an entire neighborhood. They knock on each door (send a probe), note which buildings are occupied (host discovery), test each door, window, and entry point (port scanning), identify what kind of business operates inside (service detection), and note the construction style of the building (OS detection). Every piece of information they gather comes from how the building **responds** to their probes.

### 🌍 Real-World Examples

- A network administrator scanning their organization's subnet to inventory all connected devices
- A penetration tester mapping an attack surface before beginning an engagement
- A security engineer verifying which ports are exposed on a public-facing server after a firewall configuration change
- A student verifying that the TCP three-way handshake (*Chapter 8, Section 5*) behaves as expected during a port scan

## 2. Why Nmap Exists

### 🎯 Network Visibility
Large networks evolve constantly — devices are added, removed, and reconfigured. Nmap gives administrators **accurate, real-time visibility** into exactly what's on their network and what's reachable.

### 🔒 Security Auditing
By revealing open ports and running services, Nmap lets security teams identify **unnecessary exposed services** that represent potential attack surfaces — and verify that hardening measures have been applied correctly.

### 🧩 Asset Discovery
Before you can secure something, you must know it exists. Nmap enables **asset discovery** — finding devices on a network that administrators may not have been aware of (unauthorized devices, forgotten servers, misconfigured IoT equipment).

### 🛠️ Troubleshooting
Nmap helps verify that services are listening on expected ports, confirming that applications are running correctly and that firewall rules are working as intended.

## 3. How Nmap Works — The Core Concept

Nmap sends **specially crafted probe packets** to target hosts and analyzes the **responses** (or lack thereof) to draw conclusions about each target.

```
 [ Nmap ] ---probe packet---> [ Target ]
                                   |
                              (responds / doesn't respond / sends RST)
                                   |
 [ Nmap ] <---response------[ Target ]
              |
         Nmap analyzes response:
         - Did it respond? (host is up)
         - What port state? (open/closed/filtered)
         - What service responded? (service detection)
         - What OS fingerprint? (OS detection)
```

The entire power of Nmap comes from the principle: **how a host responds to a probe reveals something about it** — whether it's alive, whether a port is open, what service is listening, and even what operating system it runs.

## 4. Basic Nmap Syntax

```
nmap [Scan Type] [Options] [Target]
```

| Component | Description |
|---|---|
| `nmap` | The command itself |
| `[Scan Type]` | What kind of scan to perform (e.g., `-sS`, `-sU`, `-sV`) |
| `[Options]` | Additional modifiers (timing, output format, verbosity, etc.) |
| `[Target]` | The IP address, hostname, range, or CIDR notation to scan |

### 🧠 Simplest Possible Scan

```bash
nmap 192.168.1.1
```

This performs a default scan against a single host — Nmap will determine if the host is up and scan the 1,000 most common TCP ports.

## 5. Target Specification

Nmap accepts targets in multiple formats, all covered in CIDR notation (*Chapter 5, Section 9*) and IP addressing (*Chapter 5*):

| Format | Example | Meaning |
|---|---|---|
| **Single IP** | `192.168.1.1` | One specific host |
| **Hostname** | `example.com` | A domain name (resolved via DNS) |
| **CIDR Range** | `192.168.1.0/24` | All 256 addresses in this subnet |
| **IP Range** | `192.168.1.1-50` | All IPs from .1 to .50 |
| **Wildcard** | `192.168.1.*` | All IPs in the last octet (equivalent to /24) |
| **Multiple targets** | `192.168.1.1 192.168.1.5` | Two specific hosts |
| **From file** | `-iL targets.txt` | Read targets from a text file |
| **Exclude** | `--exclude 192.168.1.5` | Scan everything except this IP |

## 6. Host Discovery

Before scanning ports, Nmap needs to determine which hosts are actually **alive** on the network. This is called **host discovery** (also called **ping scanning**).

| Option | Technique | When to Use |
|---|---|---|
| `-sn` | Ping sweep only — no port scan | Quickly find live hosts on a network |
| `-Pn` | Skip host discovery — treat all hosts as up | When ICMP is blocked by a firewall |
| `-PS [ports]` | TCP SYN ping | Checking specific TCP ports for host liveness |
| `-PA [ports]` | TCP ACK ping | Bypasses some firewalls that block SYN |
| `-PU [ports]` | UDP ping | Checking UDP-based host discovery |
| `-PE` | ICMP Echo request (standard ping) | Default on local networks |

```bash
nmap -sn 192.168.1.0/24
```
> Performs a ping sweep of the entire /24 subnet — reports which hosts are online without scanning ports.

## 7. Port States

When Nmap probes a port, it classifies the result into one of six possible states:

| State | Meaning |
|---|---|
| **Open** | A service is actively listening and accepting connections on this port |
| **Closed** | No service is listening, but the port is reachable — the host responds with TCP RST |
| **Filtered** | Nmap cannot determine the port state — likely a firewall is dropping probes silently |
| **Unfiltered** | The port is reachable but Nmap can't determine open or closed (seen with ACK scans) |
| **Open\|Filtered** | Nmap cannot distinguish between open and filtered — no response received |
| **Closed\|Filtered** | Nmap cannot distinguish between closed and filtered |

> 🔑 **Open** and **Filtered** are the most important states to understand:
> - **Open** = something is listening — worth investigating further
> - **Filtered** = a firewall is likely blocking probes — the service may or may not exist

## 8. Port Scanning Techniques

Nmap supports multiple scanning techniques, each using different TCP/IP behaviors to determine port states — directly leveraging TCP flags (*Chapter 8, Section 11*) and protocol mechanics.

| Technique | Option | How It Works | Requires Root? | Use Case |
|---|---|---|---|---|
| **TCP SYN Scan** | `-sS` | Sends SYN; open=SYN-ACK received, closed=RST, filtered=no reply | Yes | **Default and most common** — fast, relatively stealthy (never completes handshake) |
| **TCP Connect Scan** | `-sT` | Completes the full TCP three-way handshake | No | When root privileges unavailable |
| **UDP Scan** | `-sU` | Sends UDP packets; open=no reply or service response, closed=ICMP unreachable | Yes | Scanning UDP services (DNS, SNMP, DHCP) |
| **TCP ACK Scan** | `-sA` | Sends ACK; used to map firewall rules, not open ports | Yes | Firewall rule mapping |
| **TCP FIN Scan** | `-sF` | Sends FIN; closed=RST, open=no reply | Yes | Stealthy, bypasses some simple firewalls |
| **NULL Scan** | `-sN` | Sends packet with no flags set | Yes | Stealthy, bypasses some simple firewalls |
| **Xmas Scan** | `-sX` | Sends FIN, URG, PSH flags | Yes | Stealthy, bypasses some simple firewalls |

### Which Scan Should I Use?

| Situation | Recommended Scan |
|---|---|
| General network reconnaissance | `-sS` |
| No root privileges | `-sT` |
| Checking UDP services | `-sU` |
| Firewall analysis | `-sA` |
| Basic host discovery | `-sn` |

> 🔑 For most real-world situations, a SYN scan (`-sS`) combined with version detection (`-sV`) provides an excellent balance of speed, accuracy, and information.

### 🧠 TCP SYN Scan in Detail

The SYN scan (`-sS`) is Nmap's **default and most widely used** technique:

```
 Open port:
 Nmap -> SYN -> Target
 Nmap <- SYN-ACK <- Target    (port is open — service is listening)
 Nmap -> RST -> Target        (Nmap resets without completing handshake)

 Closed port:
 Nmap -> SYN -> Target
 Nmap <- RST <- Target        (port is closed — no service listening)

 Filtered port:
 Nmap -> SYN -> Target
 (no response)                (firewall is dropping packets silently)
```

> 🔑 The SYN scan is called "half-open" because it **never completes the three-way handshake** (*Chapter 8, Section 5*) — making it faster and less likely to appear in application-level logs than a full connect scan.

## 9. Service and Version Detection

Simply knowing a port is open isn't always enough — Nmap can probe further to identify **what service** is listening and **which version** of that service is running.

```bash
nmap -sV 192.168.1.1
```

```
 PORT     STATE  SERVICE  VERSION
 22/tcp   open   ssh      OpenSSH 8.4p1 Ubuntu 5ubuntu1.3
 80/tcp   open   http     Apache httpd 2.4.52 ((Ubuntu))
 443/tcp  open   ssl/http Apache httpd 2.4.52 ((Ubuntu))
 3306/tcp open   mysql    MySQL 8.0.28-0ubuntu0.21.10.3
```

### 🧠 Why Version Detection Matters for Security

Knowing the **exact version** of a running service allows:
- Checking against vulnerability databases (CVEs) to see if the version is known to be vulnerable
- Identifying outdated, unpatched software
- Understanding the complete attack surface of a target

## 10. Operating System Detection

Nmap can attempt to identify the **operating system** of a target by analyzing subtle differences in how different OS TCP/IP stacks respond to specially crafted probe packets — a technique called **OS fingerprinting**.

```bash
nmap -O 192.168.1.1
```

```
 OS details: Linux 4.15 - 5.6
 Network Distance: 1 hop
```

### 🧠 How OS Fingerprinting Works

Different operating systems implement the TCP/IP stack slightly differently — variations in initial TTL values (*Chapter 13, Section 7*), TCP window sizes, and responses to unusual flag combinations create a **unique fingerprint** that Nmap compares against its database of known OS signatures.

> 🔑 OS detection requires at least one open and one closed port to produce reliable results, and root/administrator privileges to perform the necessary probes.

## 11. Nmap Scripting Engine (NSE)

The **Nmap Scripting Engine (NSE)** extends Nmap's capabilities far beyond port scanning, using **Lua scripts** to perform automated tasks against discovered hosts and services.

| NSE Category | Purpose | Example Scripts |
|---|---|---|
| **auth** | Test for default or weak credentials | `ftp-anon`, `ssh-auth-methods` |
| **default** | Run safe, general-purpose information gathering | Included with `-sC` |
| **discovery** | Gather additional information about targets | `dns-brute`, `snmp-info` |
| **exploit** | Attempt to exploit known vulnerabilities | `ms17-010` (EternalBlue) |
| **safe** | Non-intrusive, safe to run on production systems | Majority of default scripts |
| **vuln** | Check for known vulnerabilities | `vuln` category scripts |

### 🧠 Running NSE Scripts

```bash
nmap -sC 192.168.1.1              # Run default scripts
nmap --script vuln 192.168.1.1    # Run vulnerability detection scripts
nmap --script http-title 192.168.1.1  # Get HTTP page titles
nmap --script banner 192.168.1.1  # Grab service banners
```

## 12. Timing and Performance

Nmap's speed can be controlled using **timing templates** — important for balancing scan speed against detection avoidance and network impact.

| Template | Option | Speed | Use Case |
|---|---|---|---|
| Paranoid | `-T0` | Extremely slow | Maximum stealth — evading IDS |
| Sneaky | `-T1` | Very slow | Stealth-oriented |
| Polite | `-T2` | Slow | Reduces network load |
| Normal | `-T3` | Default | Balanced default |
| Aggressive | `-T4` | Fast | Fast networks, CTF, lab environments |
| Insane | `-T5` | Very fast | Maximum speed — may miss results on slow networks |

> 🔑 **T4** is commonly used in CTF competitions and controlled lab environments. **T1 or T2** are used in real penetration tests where stealth and avoiding detection by IDS/IPS (*Chapter 30*) is a priority. Never use aggressive timing against production systems without authorization.

## 13. Output Formats

Nmap supports multiple output formats, useful for saving results for later analysis or importing into other tools.

| Format | Option | Description |
|---|---|---|
| **Normal** | `-oN file.txt` | Human-readable text output |
| **XML** | `-oX file.xml` | Machine-readable XML, importable into tools like Metasploit |
| **Grepable** | `-oG file.gnmap` | Single-line-per-host format for easy grep/awk processing |
| **All formats** | `-oA basename` | Saves normal, XML, and grepable simultaneously |

```bash
nmap -sV -oA scan_results 192.168.1.0/24
```

## 14. Common Nmap Command Reference

A practical quick-reference for the most commonly used combinations:

| Command | Purpose |
|---|---|
| `nmap 192.168.1.1` | Default scan — top 1000 TCP ports |
| `nmap -sn 192.168.1.0/24` | Ping sweep — find live hosts only |
| `nmap -p 80,443,22 192.168.1.1` | Scan specific ports only |
| `nmap -p- 192.168.1.1` | Scan all 65535 ports |
| `nmap -sS 192.168.1.1` | SYN scan (default, requires root) |
| `nmap -sU -p 53,67,161 192.168.1.1` | UDP scan of common UDP ports |
| `nmap -sV 192.168.1.1` | Service and version detection |
| `nmap -O 192.168.1.1` | OS detection |
| `nmap -sC 192.168.1.1` | Run default NSE scripts |
| `nmap -A 192.168.1.1` | Aggressive — OS, version, scripts, traceroute |
| `nmap -Pn 192.168.1.1` | Skip host discovery (assume host is up) |
| `nmap -T4 192.168.1.0/24` | Fast timing on a local network |
| `nmap -oA results 192.168.1.1` | Save output in all formats |
| `nmap -v 192.168.1.1` | Verbose output |
| `nmap -sV --script vuln 192.168.1.1` | Version detection + vulnerability scripts |

### Commands You'll Use Most Often

```bash
# Discover live hosts
nmap -sn 192.168.1.0/24

# Scan the most common ports
nmap 192.168.1.10

# Service detection
nmap -sV 192.168.1.10

# OS detection
nmap -O 192.168.1.10

# Full scan
nmap -A 192.168.1.10
```

If you're just starting with Nmap, these five commands cover the majority of day-to-day use cases.

## 16. Nmap and Cybersecurity

| Security Context | Relevance |
|---|---|
| **Reconnaissance** | Nmap is a primary tool in the reconnaissance phase of any penetration test or attack — mapping hosts, open ports, services, and OS details |
| **Attack Surface Mapping** | Reveals every potential entry point on a network — open ports, running services, and service versions |
| **Vulnerability Assessment** | NSE scripts can check for specific known vulnerabilities against discovered services |
| **Firewall Rule Verification** | ACK scans and filtered port results help map what firewall rules are actually in place |
| **Defender Use** | Defenders run Nmap against their own infrastructure to verify it matches the expected baseline and identify unauthorized services |
| **IDS/IPS Detection** | Nmap scans generate recognizable traffic patterns — security monitoring tools (*Chapter 30*) detect and alert on port scanning activity |
| **Ethical and Legal Boundaries** | Running Nmap against systems you don't own or have explicit written permission to test is **illegal** in most jurisdictions — Nmap should only be used on your own systems or with explicit authorization |

> 🔑 **The golden rule of Nmap:** *Only scan networks and systems you own or have explicit, written permission to test.* Unauthorized scanning is a criminal offense in most jurisdictions. In a professional context, this authorization is documented in a **Rules of Engagement (ROE)** document agreed upon before a penetration test begins.

### Blue Team vs. Red Team Perspective

| Role | How Nmap Is Used |
|---|---|
| **Red Team / Penetration Tester** | Discover attack surfaces and identify potential entry points |
| **Blue Team / Defender** | Verify firewall rules, identify unauthorized services, and audit network exposure |
| **SOC Analyst** | Validate alerts and investigate suspicious hosts during incident response |

## 17. Troubleshooting Perspective

| Problem | Symptom | What Nmap Helps Determine | Common Finding |
|---|---|---|---|
| Service unreachable from outside | External users can't reach a service | Whether the port shows as open, closed, or filtered from the external perspective | Filtered — firewall is blocking; Closed — service not running; Open — another issue (application-level) |
| Unknown devices on network | Unexpected hosts appearing in DHCP logs | Run `-sn` ping sweep to list all live hosts and their MACs | Unauthorized device, forgotten server, or IoT device |
| Verifying firewall rules | Uncertainty about what's actually blocked | Run scans against the firewall's external interface | Ports showing open that should be filtered indicate a misconfigured rule |
| Slow scan results | Nmap taking very long to complete | Review timing template and whether `-Pn` is needed | Adjust timing, restrict port range with `-p`, add `-Pn` if hosts aren't responding to ping |

## 18. Real-World Walkthrough: Scanning a Local Network

**Scenario:** You're a network administrator wanting to audit your `192.168.1.0/24` home/lab network.

**Step 1 — Discover live hosts:**
```bash
nmap -sn 192.168.1.0/24
```
```
 Nmap scan report for 192.168.1.1
 Host is up (0.002s latency).
 MAC Address: 11:22:33:44:55:66 (Router manufacturer)

 Nmap scan report for 192.168.1.10
 Host is up (0.001s latency).
 MAC Address: AA:BB:CC:DD:EE:FF (Dell Inc.)
```

**Step 2 — Scan a specific host for open ports and services:**
```bash
nmap -sV -sC 192.168.1.10
```
```
 PORT     STATE SERVICE  VERSION
 22/tcp   open  ssh      OpenSSH 8.4p1 Ubuntu
 80/tcp   open  http     Apache httpd 2.4.52
 3306/tcp open  mysql    MySQL 8.0.28
```

**Step 3 — Run OS detection:**
```bash
nmap -O 192.168.1.10
```
```
 OS details: Linux 5.4 - 5.15
```

**Step 4 — Check for vulnerabilities:**
```bash
nmap --script vuln 192.168.1.10
```
```
 | vuln-check:
 |   CVE-2021-XXXXX:
 |     State: VULNERABLE
 |     Description: ...
```

**Step 5 — Save results:**
```bash
nmap -A -oA lab_audit 192.168.1.0/24
```

This generates `lab_audit.nmap`, `lab_audit.xml`, and `lab_audit.gnmap` — complete records of the scan for review, comparison against baselines, and import into other tools.

## 19. Common Beginner Misconceptions

| ❌ Misconception | ✅ Reality |
|---|---|
| Nmap is only used by attackers | Nmap is equally and more commonly used by **defenders**, network administrators, and security engineers for legitimate network auditing |
| A filtered port means the service definitely isn't running | Filtered means a firewall is blocking probes — the service may well be running behind it |
| Running Nmap against any IP on the Internet is legal | Unauthorized scanning is **illegal** in most jurisdictions — only scan systems you own or have explicit written permission to test |
| A default Nmap scan checks all 65535 ports | The default scans only the **1,000 most common TCP ports** — use `-p-` to scan all ports |
| Nmap can always detect the correct OS | OS detection depends on having an open and a closed port, and even then may be ambiguous or incorrect — results are not guaranteed |
| `-sS` and `-sT` produce identical results | A SYN scan never completes the handshake (stealthier); a Connect scan completes it fully (more visible in logs) |
| Nmap only works on TCP | Nmap has full UDP scanning capability with `-sU`, though UDP scans are typically slower |
| A "closed" port means there's nothing there worth investigating | Closed ports still confirm the host is reachable and can be useful for OS fingerprinting |
| Nmap automatically exploits vulnerabilities | Nmap primarily discovers and analyzes systems. Exploitation requires separate tools or specific NSE scripts used in authorized environments. |

| **Fingerprint** | A unique pattern used to identify an operating system or service |
| **Latency** | The time taken for packets to travel between source and destination |

## 21. Common Interview Questions

1. **What is Nmap and what is it used for?** A free, open-source network scanner used to discover hosts, identify open ports and services, detect operating systems, and audit network security.
2. **What is the difference between a SYN scan and a Connect scan?** A SYN scan (`-sS`) never completes the handshake (stealthier, requires root); a Connect scan (`-sT`) completes the full TCP handshake (no root required, more visible in logs).
3. **What does a "filtered" port state mean?** Probes are being silently dropped, typically by a firewall — the state of the actual port cannot be determined from outside.
4. **What does `-sV` do?** Enables service and version detection — Nmap probes open ports to identify the specific service and version running.
5. **What is the Nmap Scripting Engine (NSE)?** A Lua-based framework allowing Nmap to run scripts against discovered hosts for additional information gathering, vulnerability detection, and exploitation checks.
6. **What does `-A` do?** Enables aggressive scanning — combines OS detection (`-O`), service/version detection (`-sV`), default scripts (`-sC`), and traceroute.
7. **What does `-p-` mean?** Scan all 65535 ports instead of just the default 1,000.
8. **What is a ping sweep (`-sn`)?** A host discovery scan that determines which hosts are alive without scanning ports.
9. **Why is `-Pn` used?** To skip host discovery and treat all targets as alive — necessary when ICMP is blocked by a firewall and Nmap would otherwise assume the host is down.
10. **Is it legal to run Nmap against any IP?** No — scanning systems without explicit authorization is illegal in most jurisdictions. Nmap should only be used on systems you own or have written permission to test.

## 22. Key Takeaways

### 📋 Summary

- **Nmap** is a free, open-source network scanner that actively probes hosts to discover open ports, services, versions, and operating systems.
- It supports multiple scanning techniques — **SYN scan** (`-sS`) is the default, most common, and most efficient.
- Ports are classified as **open, closed, or filtered** based on target responses.
- **Service detection** (`-sV`) and **OS fingerprinting** (`-O`) provide critical additional detail beyond port state alone.
- The **Nmap Scripting Engine (NSE)** extends Nmap with hundreds of scripts for information gathering, vulnerability detection, and more.
- **Timing templates** (`-T0` to `-T5`) balance scan speed against stealth and network impact.
- Nmap is used by **both attackers (reconnaissance) and defenders (auditing)** — understanding it from both perspectives is essential for cybersecurity professionals.
- **Unauthorized scanning is illegal** — always ensure explicit written authorization before scanning any system you don't own.
