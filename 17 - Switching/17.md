## 1. What is Switching?

### 📘 Formal Definition

**Switching** is the process by which a **network switch** receives incoming frames at the **Data Link Layer** (*Chapter 2, Section 6*) and forwards them out the correct port toward their destination, using **MAC addresses** (*Chapter 4*) to make that decision.

### 🧠 Simple Explanation

If routing (*Chapter 16*) is about getting data from one **network** to another, switching is about getting data from one **device to another within the same network**. Once data arrives on your local network, the switch is responsible for delivering it to exactly the right device — not every device.

### 🏢 Real-World Analogy: An Internal Office Mail Room

> A switch works like an **internal office mail room**. The mail room doesn't care about letters arriving from other buildings (that's the router's job) — it only handles delivering mail to the correct desk within the building. It maintains a directory of which employee sits at which desk, and delivers each piece of mail precisely, rather than dumping a copy on every single desk.

### 🌍 Real-World Examples

- The switch in your home router silently directing traffic between your laptop, phone, and smart TV
- An enterprise switch connecting hundreds of office workstations, phones, and printers
- A data center switch forwarding traffic between hundreds of servers in the same rack

## 2. Why Switching Exists

### 🎯 Solving the Hub Problem
Before switches, **hubs** were used to connect devices on a local network. Hubs simply repeated every signal out of every port — meaning every device received every frame, regardless of the intended recipient. This wasted bandwidth, created collisions, and posed serious privacy concerns.

### ⚡ Efficient, Targeted Delivery
Switches solve this by forwarding frames **only to the specific port** where the intended recipient is connected — dramatically reducing unnecessary traffic and collisions.

### 📈 Scalability
Because switches don't flood every frame to every device, they scale far more efficiently as the number of devices on a network grows.

### 🔒 Basic Traffic Isolation
Switches provide a basic level of traffic privacy — under normal circumstances, a device can only see its own traffic plus broadcasts, not the unicast traffic between other devices.

## 3. What is a Switch?

A **network switch** is a dedicated Data Link Layer device that connects multiple devices within the same network, forwarding frames intelligently based on MAC addresses.

```
                        [ Switch ]
                       /    |    \
                      /     |     \
               [ PC-A ] [ PC-B ] [ Printer ]
               Port 1   Port 2    Port 3
```

### 🧠 Key Characteristics

| Feature | Description |
|---|---|
| **Layer** | Primarily Data Link Layer (Layer 2), though Layer 3 switches also exist (Section 11) |
| **Addressing** | Uses MAC addresses to make forwarding decisions |
| **Traffic** | Forwards only to the correct port, not all ports |
| **Speed** | Full-duplex — devices can send and receive simultaneously |
| **Broadcast domains** | All ports on a switch share one broadcast domain by default (changed by VLANs — *Chapter 18*) |

---

## 4. How Switches Forward Frames

When a frame arrives at a switch port, the switch follows a precise decision process:

```
 Frame arrives on Port 1:
 [ Source MAC: AA:BB:CC:DD:EE:FF | Destination MAC: 11:22:33:44:55:66 ]
         |
         v
 Step 1: Learn  -> Record: AA:BB:CC:DD:EE:FF is reachable via Port 1
         |
         v
 Step 2: Look up destination MAC (11:22:33:44:55:66) in MAC table
         |
         ┌────────────────────┬──────────────────────────────┐
         │ Found in table     │ Not found in table           │
         │         │          │                │             │
         │         v          │                v             │
         │  Forward to        │  Flood to all ports          │
         │  specific port     │  EXCEPT the incoming port    │
         └────────────────────┴──────────────────────────────┘
```

## 5. MAC Address Table (CAM Table)

The **MAC address table** (also called the **CAM table** — Content Addressable Memory) is the switch's internal lookup database mapping MAC addresses to the specific port they were learned on.

```
 MAC Address          | Port   | VLAN | Age (seconds)
 ---------------------|--------|------|---------------
 AA:BB:CC:DD:EE:FF     | Port 1 | 1    | 0
 11:22:33:44:55:66     | Port 2 | 1    | 45
 FF:EE:DD:CC:BB:AA     | Port 3 | 1    | 120
```

### 🧠 Key Properties

- Entries are **temporary** — they age out (typically after 5 minutes of inactivity) and are removed.
- If a device moves to a different port, the switch automatically updates the CAM table when it sees traffic from the device on the new port.
- The table has a **finite size** — exceeding it forces the switch to flood unknown traffic (relevant in security, Section 15).
- **Multiple MACs** can be learned on a single port if a hub or another switch is connected to it.

---

## 6. MAC Learning in Detail

MAC learning is the automatic process by which a switch builds its MAC address table by observing **source MAC addresses** on incoming frames.

```
 [ Initially, MAC table is empty ]

 Frame arrives on Port 1 from PC-A (MAC: AA:BB:CC:DD:EE:FF)
 -> Switch records: AA:BB:CC:DD:EE:FF = Port 1

 Frame arrives on Port 2 from PC-B (MAC: 11:22:33:44:55:66)
 -> Switch records: 11:22:33:44:55:66 = Port 2

 PC-A sends a frame to PC-B:
 -> Switch looks up 11:22:33:44:55:66 -> Port 2
 -> Forwards ONLY to Port 2
 -> PC-C on Port 3 never sees this frame
```

> 🔑 MAC learning is entirely **automatic and passive** — the switch learns simply by observing normal traffic, requiring no configuration. This same mechanism is what **MAC flooding attacks** exploit (*Chapter 4, Section 7* and Section 15 below).

---

## 7. The Three Forwarding Behaviors

A switch uses one of three forwarding behaviors depending on the destination MAC address:

| Behavior | When It Occurs | What Happens |
|---|---|---|
| **Unicast Forwarding** | Destination MAC is in the CAM table | Frame sent out the single known port only |
| **Flooding** | Destination MAC is NOT in the CAM table | Frame sent out ALL ports except the incoming port |
| **Filtering** | Source and destination MAC are on the same port | Frame discarded — the destination is on the same segment, no forwarding needed |

### 🧠 Why Flooding Happens

Flooding isn't a malfunction — it's a normal fallback for frames with **unknown destination MACs**. It's temporary: once the destination device sends any frame back, its MAC is learned and future frames are forwarded precisely.

## 8. Switching vs. Routing — Full Comparison

| Feature | Switching | Routing |
|---|---|---|
| **OSI Layer** | Data Link Layer (Layer 2) | Network Layer (Layer 3) |
| **Addresses Used** | MAC addresses | IP addresses |
| **Scope** | Within the same network/subnet | Between different networks/subnets |
| **Device** | Switch | Router |
| **Speed** | Generally faster (simpler decisions) | Slightly slower (more complex logic) |
| **Broadcast Handling** | Forwards broadcasts within the domain | Blocks broadcasts — each interface is a separate broadcast domain |

> 🔑 **Key insight:** A packet traveling from your device to a website uses **both** switching (to get from your device to the router within your local network) **and** routing (to get from your router across the Internet to the destination server).

---

## 9. Hubs vs. Switches — Why Switches Won

| Feature | Hub | Switch |
|---|---|---|
| **OSI Layer** | Physical Layer (Layer 1) | Data Link Layer (Layer 2) |
| **Intelligence** | None — repeats all traffic to all ports | Intelligent — forwards to the correct port only |
| **Collisions** | Entire hub is one collision domain | Each port is its own collision domain |
| **Privacy** | None — all devices see all traffic | Each device only sees its own traffic (plus broadcasts) |
| **Bandwidth** | Shared across all ports | Full bandwidth per port (full-duplex) |
| **Status** | Effectively obsolete | Universal standard |

> 🔑 Hubs are essentially extinct in modern networking — understanding them is important primarily because **their behavior** (flooding all traffic to all ports) is what attackers attempt to **recreate** through MAC flooding attacks (Section 15).

## 10. Spanning Tree Protocol (STP)

### 🎯 The Problem: Switching Loops

In a network with **redundant switch connections** (which is good for reliability), broadcast frames could loop indefinitely — each switch forwarding the broadcast out all ports, creating an ever-expanding storm of traffic that crashes the entire network.

```
 [ Switch A ]---[ Switch B ]
       \               /
        [ Switch C ]---

 Broadcast sent -> A forwards to B and C -> B forwards to A and C ->
 C forwards to A and B -> infinite loop -> network meltdown
```

### 🎯 The Solution: STP

**Spanning Tree Protocol (STP)** prevents loops by **logically blocking** redundant links while keeping them physically available as standby backups. Only one active path exists at any time — but if that path fails, STP automatically activates a standby link.

```
 [ Switch A ]---[ Switch B ]
       \               /
        [ Switch C ]
        (one link blocked by STP, but ready to activate if needed)
```

> 🔑 Modern networks commonly use **RSTP (Rapid Spanning Tree Protocol)**, which converges much faster than the original STP when a topology change occurs.

### Why STP Matters

Without STP, a single switching loop can create:

- Broadcast storms
- MAC table instability (MAC flapping)
- Duplicate frames
- Complete network outages

STP is one of the most important protections in Layer 2 networks.

## 11. Layer 2 vs. Layer 3 Switches

| Type | What It Does | Common Use Case |
|---|---|---|
| **Layer 2 Switch** | Forwards frames using MAC addresses only — standard switch behavior | Connecting end devices within a single subnet |
| **Layer 3 Switch** | Combines switching (Layer 2) with routing (Layer 3) capabilities — can route between VLANs/subnets internally | Enterprise networks, routing between VLANs without a dedicated router |

### 🧠 Real-World Example

In an enterprise network, a **Layer 3 switch** in the core of the network can both switch traffic within each VLAN (*Chapter 18*) and route traffic **between** those VLANs — all within the same device, at hardware speed, rather than requiring a separate router for inter-VLAN routing.

## 12. Port Types on a Switch

| Port Type | Purpose |
|---|---|
| **Access Port** | Connects to an end device (PC, printer, phone) — carries traffic for one VLAN only |
| **Trunk Port** | Connects to another switch or router — carries traffic for multiple VLANs simultaneously using VLAN tagging (*Chapter 18*) |
| **Uplink Port** | A high-speed port connecting the switch to upstream infrastructure (often a router or core switch) |

> 🔑 Understanding access vs. trunk ports is foundational before studying VLANs in *Chapter 18*, as the distinction between them is central to how VLANs are configured and maintained.

## 13. Switch Port Security

**Port security** is a feature on managed switches that limits and controls which devices can connect to a specific switch port.

| Feature | Description |
|---|---|
| **MAC Limit** | Restricts how many MAC addresses can be learned on a single port |
| **Sticky MAC** | Automatically learns and permanently binds the first MAC address seen on a port |
| **Violation Actions** | When a violation is detected: shutdown (disable port), restrict (drop frames, log), or protect (silently drop frames) |

```
 Port security configured on Port 1:
 Max MACs: 1
 Sticky: Enabled
 Violation: Shutdown

 PC-A connects to Port 1 -> MAC learned and bound
 Attacker connects different device to Port 1 -> Violation detected -> Port shutdown
```

> 🔑 Port security is one of the primary defenses against **MAC flooding attacks** — by limiting how many MACs can be learned per port, it prevents attackers from overwhelming the CAM table.

## 14. How This Works in Real Life

| Environment | Switching in Practice |
|---|---|
| **Home Networks** | The switch integrated into your home router silently handles all local traffic between connected devices |
| **Enterprise Networks** | Hierarchical switching designs (access, distribution, core layers) connect thousands of devices with redundant, high-speed paths managed by STP and RSTP |
| **Data Centers** | High-density, high-speed switches connect hundreds of servers within racks and between racks, often using modern protocols that go beyond traditional STP |
| **Cloud Environments** | Virtual switches running entirely in software handle switching between virtual machines on the same physical host |
| **SOC** | Analysts examine switching-layer logs and alerts (e.g., port security violations, STP topology changes) as potential indicators of network attacks or unauthorized device connections |

## 15. Switching and Cybersecurity

| Security Context | Relevance |
|---|---|
| **MAC Flooding** | Attacker floods the CAM table with fake MAC addresses until it overflows, forcing the switch to flood all traffic to all ports — effectively turning the switch into a hub and enabling passive sniffing (*Chapter 24*) |
| **MAC Spoofing** | Impersonating another device's MAC address to receive its traffic or bypass port security (*Chapter 4, Section 12*) |
| **ARP Poisoning** | Exploits the relationship between switching and ARP to redirect traffic (*Chapter 12, Section 11* and *Chapter 25*) |
| **STP Attacks** | An attacker sends crafted STP messages to make their device become the root bridge, potentially gaining control over traffic paths within the network |
| **VLAN Hopping** | Exploiting trunk port misconfigurations to access traffic on VLANs the attacker shouldn't be on (*Chapter 18*) |
| **Port Security** | A direct defense — limits MAC addresses per port, detects unauthorized device connections (Section 13) |
| **Dynamic ARP Inspection (DAI)** | Works at the switching layer to validate ARP packets against trusted DHCP bindings (*Chapter 12, Section 11*) |

> 🔑 The switching layer is one of the most commonly targeted layers in **internal network attacks** — because once an attacker is physically or logically on the local network, Layer 2 attacks like MAC flooding and ARP poisoning can be highly effective without requiring any Internet-facing exposure.

## 16. Troubleshooting Perspective

| Problem | Symptom | What Engineers Check First | Common Fix |
|---|---|---|---|
| Device can't communicate with others on the same network | No local connectivity despite correct IP config | Switch port status, MAC table for the device's MAC | Check cable, verify port is active, confirm MAC is being learned |
| Broadcast storm / network meltdown | Entire network becomes extremely slow or unresponsive | STP status, topology changes, duplicate frames in capture | Identify and fix the switching loop, verify STP configuration |
| Port security violation | Device suddenly loses connectivity; port shows as err-disabled | Port security logs, whether a different/unexpected device was connected | Identify unauthorized device, re-enable port, update security policy |
| Intermittent connectivity | Connections drop and recover repeatedly | Whether the device's MAC is appearing on multiple ports (potential MAC flapping) | Investigate loop or duplicate MAC, check STP convergence |

## 17. Real-World Walkthrough: Two Devices Communicating on the Same Network

1. PC-A (`192.168.1.10`, MAC: `AA:AA:AA:AA:AA:AA`) wants to send data to PC-B (`192.168.1.20`, MAC: `BB:BB:BB:BB:BB:BB`) on the same switch.
2. PC-A first resolves PC-B's MAC address using **ARP** (*Chapter 12*) — sending an ARP broadcast, which the switch floods to all ports.
3. PC-B responds with its MAC address; PC-A now has the full destination MAC.
4. PC-A sends a frame: Source MAC `AA:AA:AA:AA:AA:AA`, Destination MAC `BB:BB:BB:BB:BB:BB`, arriving on **Port 1**.
5. The switch **records** PC-A's MAC → Port 1 in the CAM table (MAC learning).
6. The switch looks up `BB:BB:BB:BB:BB:BB` in the CAM table — if found, it forwards the frame **only to Port 2** (PC-B's port). If not found, it floods to all ports except Port 1.
7. PC-B receives the frame. PC-C, connected on Port 3, **never sees this frame** — the switch delivered it precisely and privately.

## 18. Common Beginner Misconceptions

| ❌ Misconception | ✅ Reality |
|---|---|
| Switches and routers do the same job | Switches operate within a network (Layer 2, MAC addresses); routers operate between networks (Layer 3, IP addresses) — distinct and complementary roles |
| A switch always knows where every device is | When a MAC isn't in the CAM table, the switch **floods** the frame — it learns over time, not instantly |
| Flooding means the switch is broken | Flooding is normal behavior for unknown destinations — not a malfunction |
| STP slows down the network unnecessarily | STP trades a small amount of overhead for **preventing catastrophic switching loops** — a necessary trade-off |
| A home router's built-in switch is the same as a managed enterprise switch | Home switches are unmanaged with minimal features; enterprise managed switches support VLANs, port security, STP tuning, and detailed logging |
| MAC flooding only works in theory | It's a practical, real-world attack technique requiring only basic software tools readily available |
| A Layer 3 switch fully replaces a router | Layer 3 switches excel at inter-VLAN routing within a network but typically lack the WAN interfaces and advanced features of dedicated routers for Internet connectivity |
| Once a MAC is learned it stays in the table forever | CAM table entries have an **aging timer** — unused entries are automatically removed after a set period |
| Switches completely eliminate broadcasts | Switches forward broadcasts within the broadcast domain; only routers stop broadcasts between networks |

## 19. Common Terminology Glossary

| Term | Definition |
|---|---|
| **Switching** | Forwarding frames within a network based on MAC addresses |
| **Switch** | A Data Link Layer device that intelligently forwards frames to specific ports |
| **CAM Table** | Content Addressable Memory — the switch's MAC-to-port lookup table |
| **MAC Learning** | The automatic process of building the CAM table from observed traffic |
| **Unicast Forwarding** | Sending a frame only to the single port where the destination MAC is known |
| **Flooding** | Sending a frame to all ports except the incoming port when the destination MAC is unknown |
| **Filtering** | Discarding a frame when source and destination are on the same port |
| **Hub** | An obsolete Layer 1 device that repeats all signals to all ports |
| **Full-Duplex** | Simultaneous two-way communication — no collision domains per port |
| **Collision Domain** | A network segment where simultaneous transmissions cause collisions |
| **Broadcast Domain** | The scope of devices that receive broadcast frames |
| **STP** | Spanning Tree Protocol — prevents switching loops by logically blocking redundant links |
| **RSTP** | Rapid Spanning Tree Protocol — faster-converging successor to STP |
| **Root Bridge** | The switch elected as the center of the STP topology |
| **Access Port** | A switch port connected to a single end device, carrying one VLAN |
| **Trunk Port** | A switch port carrying multiple VLANs between switches or to a router |
| **Layer 3 Switch** | A switch with integrated routing capabilities |
| **Port Security** | A switch feature limiting which/how many MAC addresses can use a port |
| **Sticky MAC** | A port security option that permanently binds a learned MAC address |
| **MAC Flooding** | An attack flooding the CAM table to force the switch to behave like a hub |
| **VLAN Hopping** | An attack exploiting trunk port misconfigurations to access unauthorized VLANs |
| **MAC Flapping** | A MAC address appearing on multiple switch ports unexpectedly |
| **Broadcast Storm** | Excessive broadcast traffic that consumes network resources |
| **Unknown Unicast** | Traffic sent to a destination MAC not present in the CAM table |
| **Root Port** | The port used by a switch to reach the STP root bridge |
| **Designated Port** | The port chosen to forward traffic on a network segment in STP |

## 20. Common Interview Questions

1. **What is switching and at which OSI layer does it operate?** Forwarding frames to specific ports based on MAC addresses, operating at the Data Link Layer (Layer 2).
2. **What is a CAM table and how is it built?** The switch's MAC-to-port mapping database, built automatically by observing source MAC addresses on incoming frames.
3. **What are the three forwarding behaviors of a switch?** Unicast forwarding (known destination), flooding (unknown destination), and filtering (source and destination on the same port).
4. **What is MAC flooding and why is it dangerous?** An attacker floods the CAM table with fake MAC addresses, causing the switch to flood all traffic to all ports — enabling passive traffic interception.
5. **What is Spanning Tree Protocol and why is it needed?** A protocol that prevents switching loops by logically blocking redundant links while keeping them available as standby paths.
6. **What's the difference between an access port and a trunk port?** An access port connects to one end device and carries one VLAN; a trunk port connects to another switch/router and carries multiple VLANs.
7. **What's the difference between a Layer 2 and a Layer 3 switch?** A Layer 2 switch only forwards frames; a Layer 3 switch also has routing capabilities to route between VLANs/subnets.
8. **What is port security and what does it defend against?** A feature limiting MAC addresses per port, defending against MAC flooding and unauthorized device connections.
9. **Why did switches replace hubs?** Switches forward traffic only to the intended destination, providing efficiency, performance, and basic privacy — hubs flooded all traffic to all ports.
10. **How does switching relate to routing?** Switching operates within a network (Layer 2); routing operates between networks (Layer 3). Both are needed for end-to-end communication.

## 21. Key Takeaways

### 📋 Summary

- **Switching** operates at the **Data Link Layer**, forwarding frames within a local network using **MAC addresses**.
- Switches build a **CAM table** through **MAC learning**, enabling precise, efficient frame forwarding.
- The three forwarding behaviors are **unicast forwarding, flooding, and filtering**.
- **Spanning Tree Protocol** prevents catastrophic switching loops in redundant topologies.
- **Layer 3 switches** combine switching and routing in a single device, enabling inter-VLAN routing without a dedicated router.
- The switching layer is a major attack surface for **MAC flooding, MAC spoofing, ARP poisoning, and STP attacks** — defended by **port security, Dynamic ARP Inspection, and DHCP snooping**.
