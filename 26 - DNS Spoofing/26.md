## 1. What is DNS Spoofing?

### 📘 Formal Definition

**DNS Spoofing** (also called **DNS Poisoning** or **DNS Cache Poisoning**) is an attack in which **forged DNS responses** are delivered to a victim, causing their device to resolve a legitimate domain name to a **malicious, attacker-controlled IP address** — redirecting the victim's traffic to a fraudulent destination without their knowledge.

### 🧠 Simple Explanation

DNS translates domain names to IP addresses (*Chapter 10*). DNS Spoofing is the act of **lying to a device about what IP address a domain name resolves to** — making the victim's browser go to the attacker's server when they type in a legitimate website's address.

### 📞 Real-World Analogy: A Corrupt Phone Directory

> Imagine the entire phone system relies on a printed directory to look up phone numbers. DNS Spoofing is like **secretly replacing the real phone number for a bank** in that directory with the attacker's number. When a customer looks up the bank's number and dials, they reach the attacker — who sounds just like the bank, asks for account details, and the customer never realizes they've been redirected.

### 🌍 Real-World Examples

- A victim types `bank.com` into their browser but gets directed to a convincing fake banking website that steals their credentials
- An attacker poisons a corporate DNS server to redirect all employee traffic for `intranet.company.com` to an attacker-controlled host
- Malware on a router modifies DNS settings to route all DNS queries through an attacker's resolver
- A nation-state actor compromises ISP-level DNS servers to redirect political opposition websites to government-controlled pages

## 2. Why DNS Spoofing Works — The Root Cause

DNS Spoofing is possible because of fundamental trust and design issues in the original DNS protocol:

### 🔓 No Authentication in Traditional DNS

Standard DNS (*Chapter 10*) uses UDP (*Chapter 9*) with no cryptographic authentication. A DNS response is trusted simply because:
- It arrives on the correct UDP port (53)
- It contains the correct **Transaction ID** — a 16-bit number (0–65535) that matches the original query

With only 65,536 possible Transaction IDs, an attacker who can inject a response fast enough has a realistic chance of guessing the correct ID — especially since DNS was never designed with adversarial conditions in mind.

### 🔄 Trust of Received Responses

DNS resolvers (*Chapter 10, Section 4*) and client devices cache whatever DNS response they receive — they have no built-in way to verify the response came from the legitimate authoritative server rather than an attacker.

### 🌐 UDP Connectionlessness

Because DNS uses UDP (*Chapter 9*) rather than TCP's three-way handshake (*Chapter 8, Section 5*), there's no connection verification — a forged response arriving on the right port with the right Transaction ID will be accepted.

### ⚠️ Caching Amplifies the Problem

DNS clients and recursive resolvers cache responses to improve performance.

If a malicious response is accepted and cached:

- The victim continues using the fake IP
- Future DNS lookups return the poisoned result
- The attack remains effective until the record expires (TTL) or the cache is cleared

This is why DNS cache poisoning is much more impactful than a single spoofed response.

## 3. DNS Spoofing vs. DNS Cache Poisoning

These terms are closely related but describe slightly different scopes:

| Term | Scope | Effect |
|---|---|---|
| **DNS Spoofing** | A broad term for any attack injecting false DNS information | One device gets a wrong DNS answer |
| **DNS Cache Poisoning** | Specifically targeting a **DNS resolver's cache** | All clients using that resolver get wrong answers for the duration of the poisoned record's TTL |

> 🔑 **Cache poisoning is the more dangerous and scalable form** — by poisoning a recursive resolver's cache (*Chapter 10, Section 4*), a single successful attack affects **every device that uses that resolver** for as long as the poisoned record's TTL (*Chapter 10, Section 8*) remains valid — potentially hours.

```
 DNS Spoofing (one victim):
 [ Victim ] <-- forged response -- [ Attacker ]
 Only this one device is affected

 DNS Cache Poisoning (entire resolver):
 [ Resolver's Cache ] <-- poisoned -- [ Attacker ]
       |
   All clients using this resolver get wrong answers
```

## 4. Attack Vector 1: Local DNS Spoofing via MITM

The most practical and immediately achievable form for a local attacker.

### 🎯 How It Works

1. Attacker establishes a MITM position via **ARP Poisoning** (*Chapter 25*) — intercepting all traffic between the victim and the gateway.
2. DNS queries from the victim travel through the attacker's machine.
3. The attacker intercepts DNS queries and **responds with forged DNS replies** before the real DNS server can respond.
4. The victim's device caches the forged answer and connects to the attacker's malicious server instead of the real one.

```
 Normal DNS resolution:
 [ Victim ] --> DNS Query for bank.com --> [ DNS Server ]
 [ Victim ] <-- 93.184.216.34 (real IP) <-- [ DNS Server ]

 After MITM + DNS Spoofing:
 [ Victim ] --> DNS Query for bank.com --> [ Attacker ] --> (dropped/ignored)
 [ Victim ] <-- 192.168.1.99 (fake IP) <-- [ Attacker ]
 Victim connects to 192.168.1.99 (attacker's malicious server)
```

### 🧠 Why This is Especially Effective

- Requires no vulnerability in the DNS server itself
- Works against any domain, including HTTPS sites (if SSL stripping is also applied)
- The victim's browser displays `bank.com` in the address bar — the redirection is invisible without checking the IP

## 5. Attack Vector 2: DNS Cache Poisoning (Remote)

The classic, historically significant form of the attack — directly poisoning a remote DNS resolver's cache.

### 🎯 The Kaminsky Attack (2008)

The most famous DNS cache poisoning technique was discovered by Dan Kaminsky in 2008. Before his discovery, DNS cache poisoning was considered difficult because an attacker needed to guess the correct Transaction ID.

**Kaminsky's insight:** Instead of attacking a query already in flight, flood the resolver with forged responses for a **non-existent subdomain** while simultaneously providing a poisoned answer for the real domain in the **additional records section**.

```
 Attacker sends many forged responses to resolver:
 Query:    "random12345.bank.com exists?"
 Forged:   "No, but here's the authoritative answer:
            bank.com -> 10.10.10.10 (attacker IP)"
 (repeated thousands of times with different Transaction IDs,
  hoping one matches the resolver's actual query ID)
```

If even one forged response is accepted before the legitimate response arrives, the resolver's cache is poisoned for the entire domain.

### 🛡️ The Fix: Source Port Randomization

Modern DNS resolvers use **source port randomization** — randomizing not just the Transaction ID but also the UDP source port, increasing the guesswork from 65,536 possibilities to over **2 billion** — making brute-force Transaction ID guessing impractical.

## 6. Attack Vector 3: Rogue DNS Server

### 🎯 How It Works

An attacker sets up a **completely fraudulent DNS server** and tricks victims into using it as their resolver:

- Via a **rogue DHCP server** (*Chapter 11, Section 12*) providing the attacker's IP as the DNS server address
- Via malware that modifies the device's DNS configuration
- Via a compromised router that forwards DNS queries to the attacker's resolver

```
 Rogue DHCP offers:
 IP Address: 192.168.1.50
 Subnet Mask: 255.255.255.0
 Gateway: 192.168.1.1
 DNS Server: 10.10.10.10  <-- Attacker's rogue DNS server
```

Once victims use the attacker's DNS server, **every single DNS query** can be answered with forged results — the attacker has full control over all name resolution.

---

## 7. Attack Vector 4: DNS Hijacking at the Router

### 🎯 How It Works

An attacker who compromises a router (via weak credentials, unpatched vulnerabilities, or social engineering) modifies the **router's DNS configuration** — redirecting all DNS queries from every device on the network through the attacker's resolver or a forged response system.

```
 Compromised Router:
 DNS Setting changed from: 8.8.8.8 (legitimate)
 To: 10.10.10.10 (attacker's resolver)

 Effect: ALL devices on the network use attacker's DNS
```

This is commonly seen in **pharming attacks** — where malware or social engineering tricks users into visiting a router admin page and changing DNS settings.

## 8. What Attackers Do After DNS Spoofing

Once victims are resolving domain names to attacker-controlled IPs, several follow-on attacks become possible:

| Follow-On Attack | Description |
|---|---|
| **Credential Harvesting** | Host a convincing clone of the target website — victims enter credentials believing they're on the real site |
| **Malware Distribution** | Host malicious files or software updates at the spoofed address — victims download malware thinking it's legitimate software |
| **SSL Stripping** | Intercept HTTPS redirects and serve the fake site over plain HTTP — removing encryption |
| **Selective Redirection** | Only redirect specific domains (e.g., banking, email) while allowing all other traffic through normally — making the attack harder to detect |
| **Credential Relay** | Act as a transparent proxy — collect credentials and relay them to the real site, giving the attacker credentials while the victim successfully logs in and suspects nothing |

### Real-World Examples

Attackers commonly target:

- Banking websites
- Email providers
- Software update servers
- Corporate VPN portals
- Internal company applications

These services often contain valuable credentials or sensitive information.

## 9. DNS Spoofing and Phishing — The Connection

DNS Spoofing is fundamentally connected to **phishing** but represents a significantly more dangerous variant:

| Technique | Requires User Error? | Detectable? |
|---|---|---|
| **Traditional Phishing** | Yes — victim must click a malicious link or mistype a URL | Address bar shows the wrong domain — careful users notice |
| **DNS Spoofing** | No — victim types the correct, legitimate URL | Address bar shows the **correct domain** — extremely difficult to detect |

> 🔑 This is what makes DNS Spoofing so dangerous compared to traditional phishing — **the victim does nothing wrong**. They type the correct address, use bookmarks, or click legitimate links — and still arrive at the attacker's site, with the correct domain displayed in their browser's address bar.

## 10. How to Detect DNS Spoofing

### 🔍 Manual Verification

Compare DNS responses against known-good external resolvers:

```bash
# Query your current resolver:
nslookup example.com

# Query a known-good external resolver directly:
nslookup example.com 8.8.8.8

# If results differ — potential DNS spoofing
```

### 🔍 Checking SSL Certificates

Even if DNS redirects you to a fake server, it **cannot present a valid certificate** for the real domain (unless it has obtained a fraudulent certificate). The browser will display a certificate warning — a critical indicator.

```
 Browser warning:
 "Your connection is not private"
 "NET::ERR_CERT_AUTHORITY_INVALID"
 -> DO NOT proceed — this may indicate DNS Spoofing
```

### 🔍 Wireshark Detection

Apply display filter: `dns`

Look for:
- DNS responses arriving **before a query was sent** (forged responses)
- The same domain name resolving to **different IP addresses** in different responses
- DNS responses coming from an **unexpected source IP** (not the configured resolver)

```
 SUSPICIOUS in Wireshark:
 Query:    Who is bank.com?
 Reply 1:  bank.com is 192.168.1.99  <- arrives 0.001s later (attacker)
 Reply 2:  bank.com is 93.184.216.34 <- arrives 0.050s later (real server, ignored)
```

## 11. Defenses Against DNS Spoofing

| Defense | How It Works | Coverage |
|---|---|---|
| **DNSSEC** | Adds cryptographic signatures to DNS records — resolvers verify authenticity before caching | Prevents remote cache poisoning; doesn't stop MITM if resolver is compromised |
| **DNS over HTTPS (DoH)** | Encrypts DNS queries within HTTPS (*Chapter 10, Section 11*) — prevents MITM interception of DNS traffic | Prevents local MITM-based DNS spoofing |
| **DNS over TLS (DoT)** | Encrypts DNS queries using TLS (*Chapter 10, Section 11*) — prevents MITM interception | Prevents local MITM-based DNS spoofing |
| **HTTPS and HSTS** | Even if DNS is spoofed, invalid certificates prevent connecting to fake HTTPS sites without a browser warning | Limits damage for HTTPS sites; requires user to heed warnings |
| **Certificate Pinning** | Applications reject certificates not matching a pre-known value — even fraudulent certificates are rejected | Strong protection for specific apps |
| **Source Port Randomization** | Makes Transaction ID guessing astronomically harder for remote cache poisoning attacks | Effective against the Kaminsky-style attack |
| **Dynamic ARP Inspection** | Prevents the ARP Poisoning that enables local MITM DNS spoofing | Prevents the precondition for local attacks |
| **Trusted DNS Resolvers** | Use well-maintained, DNSSEC-validating resolvers | Reduces risk of poisoned resolver responses |

### Home User Best Practices

Even if enterprise protections are unavailable, home users can reduce risk by:

- Using trusted DNS providers
- Keeping routers updated
- Changing default router passwords
- Enabling DNS over HTTPS (DoH) when available
- Never ignoring browser certificate warnings


> 🔑 **No single defense is complete on its own** — the most robust protection combines DNSSEC (for resolver integrity), DoH/DoT (for transport privacy), HTTPS/HSTS (for end-user protection), and DAI (for preventing the MITM precondition).

## 12. How This Works in Real Life

| Environment | DNS Spoofing in Practice |
|---|---|
| **Home Networks** | Routers are common targets for DNS hijacking — compromised consumer routers redirect all household DNS traffic through attacker-controlled resolvers |
| **Enterprise Networks** | Internal DNS servers are high-value targets; enterprises use DNSSEC-validating resolvers and monitor for DNS anomalies; split-horizon DNS separates internal and external resolution |
| **Public Wi-Fi** | Rogue access points and MITM-capable attackers routinely perform DNS spoofing against unprotected clients |
| **Cloud Environments** | Cloud providers' internal DNS is generally well-protected; however, misconfigured DNS settings in cloud environments have caused significant outages and security incidents |
| **SOC** | DNS query logs are one of the highest-value data sources for detecting both DNS spoofing attacks and malware using DNS for C2 communication (*Chapter 10, Section 13*) |

## 13. DNS Spoofing and Cybersecurity

| Security Context | Relevance |
|---|---|
| **Credential Theft at Scale** | A poisoned resolver affects every client using it — potentially thousands of users redirected to credential-harvesting sites |
| **Malware Distribution** | Poisoned DNS enables "drive-by" malware distribution — legitimate software update domains redirected to malicious download servers |
| **Nation-State Attacks** | Government-level DNS manipulation has been documented — redirecting opposition websites, blocking certain domains, or performing targeted surveillance |
| **DNS as C2 Channel** | Malware uses DNS tunneling (*Chapter 10, Section 13*) — SOC analysts monitor DNS logs for abnormal query patterns, unusually long domain names, or high-frequency querying |
| **BGP + DNS Combined Attacks** | Sophisticated attackers combine BGP hijacking (*Chapter 16, Section 14*) with DNS manipulation for internet-scale traffic redirection |
| **Supply Chain Attacks** | Compromising a DNS provider affects all domains managed through that provider simultaneously |

## 14. Troubleshooting Perspective

| Problem | Symptom | DNS Spoofing as Potential Cause | Investigation Steps |
|---|---|---|---|
| Site loads wrong content | Correct URL typed but wrong site appears | DNS resolving to wrong IP | Compare `nslookup` against known-good resolver; check SSL certificate |
| SSL certificate warnings on trusted sites | Browser warns about invalid certificate on a normally trusted site | DNS redirecting to attacker's server without valid cert | Never proceed past cert warnings; investigate DNS configuration |
| Intermittent "wrong" pages | Some users reach real site, others reach fake | Partial DNS cache poisoning affecting some resolvers | Check resolver configuration, query multiple DNS servers |
| Malware downloaded from "legitimate" source | Known-good software source delivers malware | DNS redirecting software update domain | Verify file hashes against official values; check DNS settings |

### Useful Linux Commands

#### Check Current DNS Servers

```bash
cat /etc/resolv.conf
```

#### Query a Specific Resolver

```bash
nslookup example.com 8.8.8.8
```

#### Query Using dig

```bash
dig example.com
```

#### Flush Local DNS Cache (systemd)

```bash
sudo resolvectl flush-caches
```

These commands are commonly used when investigating DNS resolution problems.

## 15. Real-World Walkthrough: Local DNS Spoofing via ARP Poisoning

> ⚠️ For educational purposes only. Only perform in an authorized lab environment on equipment you own.

**Scenario:** Attacker on the same local network as a victim intercepts and spoofs DNS responses.

**Environment:**
- Victim: `192.168.1.10`
- Gateway/DNS: `192.168.1.1`
- Attacker: `192.168.1.99` (Kali Linux)

**Step 1 — Establish MITM via ARP Poisoning (*Chapter 25*):**
```bash
echo 1 > /proc/sys/net/ipv4/ip_forward
bettercap -iface eth0
# In bettercap:
set arp.spoof.targets 192.168.1.10
arp.spoof on
```

**Step 2 — Set up a fake web server on the attacker's machine:**
```bash
# Simple Python web server serving a fake login page:
python3 -m http.server 80
# (In practice, a convincing clone of the target site)
```

**Step 3 — Enable DNS spoofing in bettercap:**
```bash
# In bettercap console:
set dns.spoof.domains bank.com
set dns.spoof.address 192.168.1.99
dns.spoof on
```

**Step 4 — Victim types `http://bank.com` in their browser:**
```
 Victim's DNS query: "What is bank.com?"
 Normal path: Victim -> Gateway DNS -> Real answer
 With MITM:   Victim -> Attacker   -> Forged: "bank.com is 192.168.1.99"
 Victim connects to 192.168.1.99 (attacker's fake server)
```

**Step 5 — Verify in Wireshark (on monitoring device):**
```
 DNS Response: bank.com -> 192.168.1.99
 Source: 192.168.1.99 (attacker — not the real DNS server!)
```

**Step 6 — What stops this from working on HTTPS:**
The victim's browser connects to `192.168.1.99` but requests `bank.com` via HTTPS. The attacker's server presents a self-signed certificate — the browser shows:
```
 "Your connection is not private"
 Attackers might be trying to steal your information from bank.com
```
If the victim **clicks through this warning**, the attack succeeds. If they **heed the warning**, they're protected.

**Defense:** DoH/DoT prevents step 3 from working (DNS queries are encrypted). HSTS prevents step 6's click-through by enforcing HTTPS and rejecting invalid certificates automatically.

### What You Should Observe

During the lab, verify that you can identify:

- The victim's DNS query
- The forged DNS response
- The spoofed IP address
- The browser connecting to the attacker's server
- The certificate warning (for HTTPS)
- Normal behavior after enabling the appropriate defenses

Being able to recognize these artifacts is a key skill for penetration testers, incident responders, and SOC analysts.

## 16. Common Beginner Misconceptions

| ❌ Misconception | ✅ Reality |
|---|---|
| DNS Spoofing always produces a visible fake URL | The browser shows the **correct, legitimate domain name** — the spoofing happens at the IP resolution level, below what the browser displays |
| HTTPS completely prevents DNS Spoofing from being dangerous | HTTPS with a valid certificate protects content, but the victim may still be tricked by the address bar — though the certificate error is a critical warning |
| Only large, sophisticated attackers can perform DNS Spoofing | With tools like bettercap on a local network, DNS spoofing is achievable by a beginner in minutes |
| DNSSEC completely solves the DNS Spoofing problem | DNSSEC protects against remote cache poisoning but doesn't help if the resolver itself is compromised or if a MITM attack intercepts queries before they reach a DNSSEC-validating resolver |
| Changing to a different DNS server (like 8.8.8.8) prevents local DNS Spoofing | On a MITM-compromised network, the attacker intercepts **all DNS traffic** regardless of the configured resolver — unless DoH or DoT is used to encrypt the query |
| DNS Spoofing and phishing are the same attack | Phishing sends users to wrong sites through deception; DNS Spoofing sends users to wrong sites even when they navigate correctly — DNS Spoofing is transparent to the victim |
| SSL certificate warnings can be safely dismissed | A certificate warning on a normally trusted site is a **critical security indicator** — especially possible evidence of DNS Spoofing or a MITM attack |
| DNS Spoofing only affects web browsing | Any application relying on DNS, including email, software updates, VPNs, APIs, and cloud services, can be affected |

## 17. Common Terminology Glossary

| Term | Definition |
|---|---|
| **DNS Spoofing** | Delivering forged DNS responses to redirect a victim to an attacker-controlled IP |
| **DNS Cache Poisoning** | Corrupting a DNS resolver's cache with false records affecting all its clients |
| **Transaction ID** | A 16-bit identifier in DNS queries and responses used for matching — the target of guessing attacks |
| **DNSSEC** | DNS Security Extensions — adds cryptographic signatures to DNS records for authenticity verification |
| **DoH** | DNS over HTTPS — encrypts DNS queries within HTTPS to prevent interception |
| **DoT** | DNS over TLS — encrypts DNS queries using TLS to prevent interception |
| **HSTS** | HTTP Strict Transport Security — forces browsers to use HTTPS and reject invalid certificates |
| **Pharming** | A broad term for DNS-based attacks redirecting users from legitimate to fraudulent sites |
| **Rogue DNS Server** | An attacker-controlled DNS resolver used to provide false answers |
| **DNS Hijacking** | Modifying a system's or router's DNS configuration to point to a malicious resolver |
| **Split-Horizon DNS** | Providing different DNS answers to internal vs. external queries |
| **Source Port Randomization** | Randomizing the UDP source port of DNS queries to prevent Transaction ID guessing |
| **The Kaminsky Attack** | A 2008 DNS cache poisoning technique that demonstrated the severity of DNS's design flaws |
| **Credential Harvesting** | Collecting usernames and passwords via a fake but convincing login page |
| **DNS Tunneling** | Covertly encoding data in DNS traffic for exfiltration or C2 communication (*Chapter 10*) |
| **Certificate Pinning** | An application security technique that rejects certificates not matching a pre-known value |
| **Recursive Resolver** | A DNS server that performs lookups on behalf of clients |
| **Authoritative Server** | The server that holds the official DNS records for a domain |
| **Resolver Cache** | Stored DNS responses used to answer future queries quickly |
| **TTL (Time To Live)** | The lifetime of a cached DNS record before it expires |
| **DNS Record** | Information stored in DNS, such as A, AAAA, MX, CNAME, or TXT records |

## 18. Common Interview Questions

1. **What is DNS Spoofing and how does it work?** An attack that delivers forged DNS responses to redirect a victim's traffic from a legitimate domain to an attacker-controlled IP address.
2. **What is the difference between DNS Spoofing and DNS Cache Poisoning?** DNS Spoofing affects one victim's resolution; DNS Cache Poisoning poisons a resolver's cache affecting all clients using it.
3. **Why is DNS vulnerable to spoofing?** Traditional DNS uses unauthenticated UDP with only a 16-bit Transaction ID for response matching — no cryptographic verification of response authenticity.
4. **What is the Kaminsky Attack?** A 2008 DNS cache poisoning technique that exploited the small Transaction ID space to poison resolvers at scale — leading to widespread adoption of source port randomization.
5. **Why is DNS Spoofing more dangerous than traditional phishing?** The victim types the correct URL and nothing looks wrong — the address bar shows the legitimate domain name while they're actually on a fake site.
6. **What is DNSSEC and how does it help?** DNS Security Extensions add cryptographic signatures to DNS records, allowing resolvers to verify authenticity and detect forged responses.
7. **Why doesn't HTTPS fully prevent DNS Spoofing attacks?** HTTPS with a valid certificate prevents content interception, but the attacker can't get a valid certificate for the victim's domain — the browser certificate warning is the key protection layer.
8. **How does DoH prevent local DNS Spoofing?** By encrypting DNS queries within HTTPS, a MITM attacker cannot see, intercept, or forge DNS traffic between the client and the DoH server.
9. **How would you detect DNS Spoofing in a Wireshark capture?** Look for DNS responses arriving before a query, DNS answers from unexpected source IPs, or the same domain resolving to different IPs in different responses.
10. **What is the most important user-facing indicator of DNS Spoofing?** A TLS/SSL certificate warning on a site that normally has a valid certificate — this should never be dismissed.

## 19. Key Takeaways

### 📋 Summary

- **DNS Spoofing** forges DNS responses to redirect victims from legitimate domain names to attacker-controlled IP addresses — transparently, with the correct URL still displayed in the browser.
- It works because traditional DNS has **no authentication** — UDP responses are trusted based only on arriving with the correct Transaction ID.
- **DNS Cache Poisoning** is the most scalable variant — poisoning a resolver's cache affects **all clients using that resolver**.
- Attack vectors include MITM-based local spoofing, the Kaminsky remote cache attack, rogue DNS servers, and router DNS hijacking.
- Follow-on attacks include **credential harvesting, malware distribution, and traffic interception**.
- Detection relies on comparing DNS responses against known-good resolvers, watching for unexpected SSL certificate warnings, and monitoring DNS traffic in Wireshark.
- Defenses include **DNSSEC** (resolver integrity), **DoH/DoT** (transport encryption), **HTTPS/HSTS** (end-user protection), and **DAI** (preventing the MITM precondition).
