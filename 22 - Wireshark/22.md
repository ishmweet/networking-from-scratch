## 1. What is Wireshark?

### 📘 Formal Definition

**Wireshark** is a free, open-source **network protocol analyzer** (also called a **packet sniffer** or **packet analyzer**) that captures live network traffic from a network interface and displays each packet in a detailed, human-readable, layered format for analysis.

### 🧠 Simple Explanation

Wireshark is a **microscope for network traffic**. It lets you capture every single packet flowing through a network interface and inspect it in complete detail — seeing every byte, every protocol header, and every piece of data exactly as it travels across the network.

### 🔬 Real-World Analogy: An X-Ray Machine for Network Traffic

> Normal network communication happens invisibly — packets fly across cables and airwaves at millisecond speeds with no visibility to the observer. Wireshark is like putting an **X-ray machine** on the network — suddenly you can see inside every packet, reading its headers, its contents, and its protocol details layer by layer, exactly as the OSI Model (*Chapter 2*) describes.

### 🌍 Real-World Examples

- A network engineer capturing traffic to diagnose why a specific application is slow
- A security analyst examining a packet capture to investigate a suspected data breach
- A student verifying that a TCP three-way handshake (*Chapter 8, Section 5*) actually looks exactly as described in theory
- A penetration tester capturing credentials transmitted over unencrypted protocols

## 2. Why Wireshark Exists

### 🎯 Visibility into Real Traffic
Theoretical networking knowledge is essential, but seeing **real packets** is what makes concepts truly concrete. Wireshark bridges the gap between textbook theory and what actually happens on a real network.

### 🛠️ Troubleshooting
When something is wrong with a network connection or application, Wireshark lets engineers see **exactly what's happening at the packet level** — not just what applications report, but what the network is actually doing.

### 🔒 Security Analysis
Security professionals use Wireshark to detect **malicious traffic, unauthorized communications, data exfiltration**, and protocol anomalies that other tools might miss.

### 🎓 Learning and Verification
For anyone learning networking, Wireshark is invaluable — you can capture a real DNS lookup, a real TLS handshake, or a real ARP request and compare it directly to the theory covered in earlier chapters.

## 3. How Wireshark Works — The Core Concept

Wireshark uses the system's network interface in **promiscuous mode** to capture traffic.

### 🎯 Promiscuous Mode

Normally, a network interface only captures packets **addressed to its own MAC address** (plus broadcasts). In **promiscuous mode**, the interface captures **all packets** it receives, regardless of the destination MAC address — including traffic intended for other devices on the same network segment.

```
 Normal Mode:
 [ Frame for MAC: AA:BB ] arrives -> Captured ✓
 [ Frame for MAC: CC:DD ] arrives -> Discarded ✗

 Promiscuous Mode:
 [ Frame for MAC: AA:BB ] arrives -> Captured ✓
 [ Frame for MAC: CC:DD ] arrives -> Captured ✓
 [ All frames                   ] -> Captured ✓
```

### 🎯 Monitor Mode vs Promiscuous Mode

These terms are often confused:

| Mode | Purpose |
|---|---|
| **Promiscuous Mode** | Captures all Ethernet frames received by the interface |
| **Monitor Mode** | Captures raw 802.11 wireless frames before they are processed by the operating system |

> 🔑 Promiscuous Mode is primarily used on wired networks. Monitor Mode is used on wireless networks and is essential for wireless security analysis.

### 🎯 What Gets Captured

What you can actually capture depends on your network topology:
- On a **switched network** (which is virtually every modern network), you only see your own traffic, broadcasts, and multicast — not unicast traffic between other devices (unless you use port mirroring or are in a position to intercept that traffic)
- On a **wireless network in monitor mode** (*Chapter 21, Section 7*), you can capture all wireless frames in range
- On a **network with a hub** (essentially obsolete — *Chapter 17, Section 9*), you see all traffic

## 5. Capturing Traffic

### 🎯 Starting a Capture

1. Open Wireshark
2. Select the network interface to capture on (e.g., `eth0` for wired, `wlan0` for wireless)
3. Optionally set a capture filter (Section 6)
4. Click the **blue shark fin** (Start Capture) button
5. Generate or wait for network traffic
6. Click the **red square** (Stop Capture) button when done

### 🎯 Saving Captures

Captured traffic is saved in **PCAP format** (`.pcap` or `.pcapng`) — a standard file format that can be shared, reopened in Wireshark later, or analyzed by other tools (e.g., Zeek, Suricata, tcpdump).

> 🔑 **PCAP files** are widely used across the cybersecurity industry — CTF challenges, incident response investigations, and security certifications all commonly involve analyzing pre-captured PCAP files.

## 6. Capture Filters

**Capture filters** are applied **before** packets are captured — only packets matching the filter are captured at all. They use **BPF (Berkeley Packet Filter)** syntax.

### 🧠 Why Use Capture Filters?

On a busy network, capturing everything generates enormous amounts of data quickly. A targeted capture filter reduces noise to only the traffic you care about.

### 📋 Common Capture Filter Examples

| Filter | What It Captures |
|---|---|
| `host 192.168.1.10` | All traffic to or from this IP |
| `net 192.168.1.0/24` | All traffic within this subnet |
| `port 443` | Only HTTPS traffic |
| `port 53` | Only DNS traffic |
| `tcp` | Only TCP traffic |
| `udp` | Only UDP traffic |
| `not arp` | Everything except ARP |
| `host 192.168.1.10 and port 80` | HTTP traffic to/from this IP only |
| `src host 10.0.0.1` | Traffic originating from this IP |
| `dst port 22` | Traffic destined for SSH |

---

## 7. Display Filters

**Display filters** are applied **after** capture — they show or hide packets from an already-captured set without discarding them. They use Wireshark's own, more expressive filter syntax.

### 🧠 Why Use Display Filters?

You often capture a broad set of traffic, then use display filters to narrow down to specific conversations, protocols, or conditions of interest during analysis.

### 📋 Common Display Filter Examples

| Filter | What It Shows |
|---|---|
| `ip.addr == 192.168.1.10` | All traffic involving this IP |
| `ip.src == 192.168.1.10` | Traffic originating from this IP |
| `ip.dst == 8.8.8.8` | Traffic destined for this IP |
| `tcp.port == 443` | All HTTPS traffic |
| `dns` | All DNS packets |
| `http` | All HTTP packets |
| `arp` | All ARP packets |
| `tcp.flags.syn == 1` | Only TCP SYN packets |
| `tcp.flags.syn == 1 and tcp.flags.ack == 0` | Only the initial SYN (not SYN-ACK) |
| `http.request.method == "POST"` | Only HTTP POST requests |
| `tls.handshake.type == 1` | Only TLS ClientHello messages |
| `frame contains "password"` | Any frame containing the word "password" |
| `!(arp or dns or icmp)` | Exclude ARP, DNS, and ICMP noise |

> 🔑 **Capture filters vs. display filters** is a common source of beginner confusion. Remember: **capture filters** decide what gets saved; **display filters** decide what you see from what was saved.

## 8. Reading a Packet — Anatomy of a Capture

Selecting a packet in the Packet List Panel and expanding the Packet Details Panel reveals the **full OSI-layered breakdown**:

```
 ▼ Frame 3: 74 bytes on wire, 74 bytes captured
     [Arrival Time: ...]
     [Frame Number: 3]

 ▼ Ethernet II                        <- Layer 2 (Data Link)
     Destination: 11:22:33:44:55:66   <- Destination MAC
     Source: AA:BB:CC:DD:EE:FF        <- Source MAC
     Type: IPv4 (0x0800)

 ▼ Internet Protocol Version 4        <- Layer 3 (Network)
     Source Address: 192.168.1.10     <- Source IP
     Destination Address: 93.184.216.34 <- Destination IP
     Time to Live: 64                 <- TTL (Chapter 13)
     Protocol: TCP (6)

 ▼ Transmission Control Protocol      <- Layer 4 (Transport)
     Source Port: 51234               <- Ephemeral port (Chapter 7)
     Destination Port: 443            <- HTTPS (Chapter 14)
     Sequence Number: 0
     Flags: SYN                       <- Beginning a connection (Chapter 8)
     Window Size: 64240
```

> 🔑 This is the OSI Model (*Chapter 2*) and TCP/IP Model (*Chapter 3*) made **completely tangible** — each protocol header you've studied throughout this knowledge base appears here exactly as described, in a real captured packet.

## 9. Following a Stream

The **Follow Stream** feature reconstructs the complete, bidirectional conversation between two endpoints from the individual packets in a capture — reassembling a TCP session, UDP exchange, or HTTP request/response into a single readable view.

```
 Right-click a packet -> Follow -> TCP Stream

 Result: Complete conversation shown as human-readable text:

 GET /index.html HTTP/1.1
 Host: example.com
 User-Agent: Mozilla/5.0

 HTTP/1.1 200 OK
 Content-Type: text/html
 ...
 <html>...</html>
```

### 🧠 Why This Matters for Security

Following a TCP/HTTP stream lets analysts instantly see the **full content of a conversation** — what was requested, what was returned, what data was transmitted — without having to manually piece together hundreds of individual packets.

## 10. Protocol Dissectors

Wireshark includes hundreds of built-in **protocol dissectors** — modules that understand specific protocols and automatically decode packet contents into labeled, human-readable fields.

This is why Wireshark can display a DNS packet as:

```
 Domain Name System (query)
   Transaction ID: 0xabcd
   Flags: 0x0100 Standard query
   Questions: 1
   Queries:
     example.com: type A, class IN
```

Rather than just raw bytes — Wireshark knows the DNS protocol format (*Chapter 10*) and applies the appropriate dissector automatically.

## 11. Wireshark and the OSI Model — Layers in Practice

Wireshark is the single best tool for **observing the OSI Model in action** on real traffic.

| OSI Layer | What You See in Wireshark |
|---|---|
| **Layer 7 (Application)** | HTTP requests/responses, DNS queries/answers, DHCP messages |
| **Layer 6 (Presentation)** | TLS handshakes, encryption records |
| **Layer 5 (Session)** | TCP session establishment and teardown |
| **Layer 4 (Transport)** | TCP/UDP headers, port numbers, flags, sequence numbers |
| **Layer 3 (Network)** | IP headers, source/destination IPs, TTL, protocol field |
| **Layer 2 (Data Link)** | Ethernet headers, source/destination MAC addresses |
| **Layer 1 (Physical)** | Not directly visible (Wireshark captures above the physical layer) |

## 14. Wireshark and Cybersecurity

| Security Context | Relevance |
|---|---|
| **Cleartext Credential Detection** | Wireshark can immediately reveal usernames and passwords transmitted over unencrypted protocols (HTTP, FTP, Telnet, plain SMTP) — a compelling demonstration of why encryption matters |
| **ARP Poisoning Detection** | Anomalous ARP replies mapping one IP to multiple MACs are immediately visible in a Wireshark capture (*Chapter 25*) |
| **DNS Spoofing Detection** | Unexpected DNS responses with unusual IPs or timing anomalies are detectable through capture analysis (*Chapter 26*) |
| **Malware Beaconing** | Regular, interval-based outbound connections from a compromised host to a C2 server produce distinctive patterns visible in captures |
| **Data Exfiltration** | Unusually large outbound flows or encoding patterns in DNS/ICMP traffic can indicate exfiltration attempts |
| **TLS Inspection** | While TLS content is encrypted, Wireshark can still reveal the server certificate, cipher suite, and SNI (Server Name Indication) — metadata useful for threat intelligence |
| **PCAP in CTF Competitions** | Wireshark is the primary tool for network forensics challenges in Capture The Flag competitions |
| **Incident Response** | PCAP files are often collected during security incidents and analyzed to reconstruct attacker activity |
| **Threat Hunting** | Analysts examine packet captures to identify unusual communications, suspicious protocols, or unauthorized destinations |

> 🔑 The single most important demonstration for any networking student: capture HTTP (not HTTPS) traffic to a test site and use "Follow TCP Stream" — seeing a plaintext username and password in the capture immediately explains why HTTPS (*Chapter 14/15*) and encryption exist. This exercise makes abstract security concepts viscerally real.

## 15. Troubleshooting Perspective

| Problem | Symptom | What Engineers Check in Wireshark | Common Finding |
|---|---|---|---|
| Slow application performance | Noticeable lag in a network application | TCP retransmissions, window size, round-trip time | High retransmission rate indicating packet loss on the path |
| Connection failing silently | Application can't connect, no clear error | Whether SYN packets receive a SYN-ACK or RST in response | RST response indicating the port is closed or blocked by firewall |
| DNS resolution failing | Sites unreachable by name, reachable by IP | Whether DNS queries are being sent and replies received | No replies — DNS server unreachable; wrong replies — potential spoofing |
| Intermittent connectivity | Works then stops, then works again | TCP retransmissions, duplicate ACKs, ICMP unreachable messages | Underlying network instability or MTU issues |

## 16. Real-World Walkthrough: Capturing and Analyzing an HTTP Request

> Note: This walkthrough uses plain HTTP (not HTTPS) for demonstration purposes, to show visible content in the capture.

1. Open Wireshark, select your active network interface, and start a capture.
2. Apply a capture filter: `port 80` — to capture only HTTP traffic.
3. Open a browser and visit an HTTP website (or use `curl http://example.com`).
4. Stop the capture in Wireshark.
5. Apply the display filter: `http` — to show only HTTP packets.
6. In the **Packet List Panel**, you see the HTTP GET request and the server's 200 OK response.
7. Click the GET request packet — the **Packet Details Panel** expands showing:
   - **Ethernet Layer:** Your MAC → Router MAC
   - **IP Layer:** Your IP → Server IP
   - **TCP Layer:** Source port (ephemeral) → Destination port 80, SYN flag set
   - **HTTP Layer:** `GET /index.html HTTP/1.1`, `Host: example.com`, `User-Agent: ...`
8. Right-click the HTTP packet → **Follow → TCP Stream**
9. The full request and response appear in human-readable text — including response headers, status code `200 OK`, and the HTML content of the page.
10. Every protocol layer studied in this knowledge base (Ethernet, IP, TCP, HTTP) is visible and labeled here in a single real packet.

### What You Should Notice

When analyzing the capture, try to identify:

- Source MAC Address
- Destination MAC Address
- Source IP Address
- Destination IP Address
- Source Port
- Destination Port
- Protocol Used
- HTTP Method
- HTTP Status Code

Being able to identify these fields quickly is one of the most important Wireshark skills for networking and cybersecurity professionals.

## 17. Common Beginner Misconceptions

| ❌ Misconception | ✅ Reality |
|---|---|
| Wireshark captures all traffic on the network | On a switched network, it only captures your own traffic, broadcasts, and multicast — not unicast between other devices (without port mirroring or special positioning) |
| Wireshark can decrypt HTTPS traffic automatically | TLS/SSL encrypted traffic appears as opaque encrypted bytes — decryption requires the session keys, which Wireshark can use if specifically provided |
| Capture filters and display filters work the same way | Capture filters use BPF syntax and decide what gets saved; display filters use Wireshark syntax and decide what's shown from the saved data |
| Using Wireshark on a network without permission is legal | Capturing traffic on networks you don't own or have explicit permission to monitor is **illegal** in most jurisdictions |
| Wireshark only shows data you send | In promiscuous mode on an appropriate network segment, Wireshark captures all frames received by the interface |
| Wireshark is only for advanced experts | Wireshark is extremely beginner-friendly for basic capture and display — its power grows with your networking knowledge |
| PCAP files are a Wireshark-specific format | PCAP is a widely used, open standard format used by many tools (tcpdump, Zeek, Suricata, etc.) |
| More packets always means a problem | Many networks generate thousands of normal packets every second; context matters more than volume |

## 19. Common Interview Questions

1. **What is Wireshark and what is it used for?** A free, open-source network protocol analyzer used to capture and analyze network traffic for troubleshooting, security analysis, and learning.
2. **What is promiscuous mode and why does Wireshark use it?** A NIC mode that captures all received packets regardless of destination MAC — Wireshark uses it to see traffic beyond its own device's communications.
3. **What is the difference between a capture filter and a display filter?** A capture filter (BPF syntax) decides what gets captured; a display filter (Wireshark syntax) decides what's shown from an already-captured set.
4. **What file format does Wireshark use to save captures?** PCAP (`.pcap` or `.pcapng`) — a standard format also used by tcpdump and other tools.
5. **Why can't Wireshark see all traffic on a switched network?** Switches forward unicast frames only to the intended destination port — Wireshark only sees traffic addressed to its own MAC plus broadcasts and multicast.
6. **What is the "Follow TCP Stream" feature?** It reconstructs a complete TCP conversation from individual packets, showing the full bidirectional exchange in readable form.
7. **How can Wireshark be used to detect ARP poisoning?** By looking for ARP replies where the same IP address is mapped to multiple different MAC addresses.
8. **Can Wireshark decrypt HTTPS traffic?** Not by default — TLS content appears encrypted. Decryption requires the session keys (e.g., via browser key export).
9. **What display filter shows only TCP SYN packets?** `tcp.flags.syn == 1 and tcp.flags.ack == 0`
10. **Why is Wireshark relevant to cybersecurity?** It reveals cleartext credentials on unencrypted protocols, detects ARP/DNS anomalies, identifies malware beaconing, and provides packet-level evidence during incident response.

## 20. Key Takeaways

### 📋 Summary

- **Wireshark** is a free, open-source network protocol analyzer that captures and displays network traffic in detailed, layered, human-readable form.
- It uses **promiscuous mode** to capture all frames received by an interface, not just those addressed to its own MAC.
- **Capture filters** (BPF syntax) determine what gets captured; **display filters** (Wireshark syntax) determine what's shown from captured data — these are distinct mechanisms with different syntax.
- The three-panel interface — **Packet List, Packet Details, Packet Bytes** — provides summary, layered detail, and raw bytes for every packet.
- **Follow Stream** reconstructs complete TCP/HTTP conversations from individual packets.
- Wireshark makes the **OSI and TCP/IP models tangible** — every protocol and header covered throughout this knowledge base is visible in real captured traffic.
- From a security perspective, Wireshark reveals **cleartext credentials, ARP anomalies, DNS spoofing, malware beaconing**, and is the primary tool for network forensics and PCAP-based CTF challenges.
