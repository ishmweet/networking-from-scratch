## 1. What is a VLAN?

### 📘 Formal Definition

A **VLAN (Virtual Local Area Network)** is a **logical subdivision of a physical network**, created at the Data Link Layer (*Chapter 2, Section 6*), that groups devices into separate, isolated broadcast domains regardless of their physical location or which physical switch port they are connected to.

### 🧠 Simple Explanation

A single physical switch can be divided into multiple **completely separate virtual networks** using VLANs — as if you sliced one switch into several independent switches, each invisible to the others by default.

### 🏢 Real-World Analogy: Office Floors in One Building

> Imagine a large office building where Sales, Engineering, HR, and Guest Wi-Fi all share the same physical floors and network cables. VLANs are like installing **invisible walls** between each department — they occupy the same physical space and use the same physical infrastructure, but can't directly access each other's conversations without going through a controlled gateway. A device in Sales can't just reach a device in Engineering any more than a visitor using Guest Wi-Fi can access internal systems — even though they're all physically in the same building on the same switches.

### 🌍 Real-World Examples

- Separating employee traffic from guest Wi-Fi on the same physical access point
- Isolating a hospital's medical devices onto their own VLAN away from staff workstations
- Dividing a university network into separate VLANs for students, faculty, administration, and IoT devices

## 2. Why VLANs Exist

### 🔒 Security Through Segmentation
By default, all devices on a switch share the same broadcast domain and can potentially communicate freely. VLANs enforce **logical boundaries** — traffic from one VLAN cannot reach another without explicitly passing through a router or Layer 3 device, where access control rules can be applied.

### 📉 Reducing Broadcast Traffic
Recall from Chapter 17 that broadcasts (*Chapter 4, Section 6*) are forwarded to every device in a broadcast domain. VLANs shrink broadcast domains to only the members of each VLAN, **dramatically reducing unnecessary broadcast traffic** on large networks.

### 🏗️ Logical Organization Independent of Physical Location
Without VLANs, grouping devices by function (e.g., all finance computers together) would require them to be physically near each other and on the same switch. VLANs allow **logical grouping regardless of physical location** — a Finance VLAN can span multiple floors, buildings, or switches.

### 💰 Cost Efficiency
VLANs allow one physical switch infrastructure to serve multiple logically separate networks, eliminating the need to purchase, install, and maintain separate physical switches for each department or function.

### 🧩 Simplifying Network Management
Changes to network membership (e.g., moving an employee to a different department) become a **software configuration change** (reassigning a port to a different VLAN) rather than a physical cable move.

## 3. How VLANs Work — The Core Concept

By default, every port on a switch belongs to **VLAN 1** (the default VLAN). All devices connected to any port are in the same broadcast domain and can communicate freely.

```
 [ Default: No VLANs configured ]

 Switch:
 Port 1 -- PC-A  ┐
 Port 2 -- PC-B  ├── All in one broadcast domain (VLAN 1)
 Port 3 -- PC-C  ┘
```

With VLANs configured, the switch logically separates ports into isolated groups:

```
 [ With VLANs configured ]

 Switch:
 Port 1 -- PC-A  ┐ VLAN 10 (Sales)      ─┐
 Port 2 -- PC-B  ┘                        ├── Isolated from each other
 Port 3 -- PC-C  ┐ VLAN 20 (Engineering) ─┘
 Port 4 -- PC-D  ┘

 PC-A can reach PC-B (same VLAN 10)
 PC-A CANNOT reach PC-C or PC-D (different VLAN 20)
```

The switch enforces this isolation — frames from VLAN 10 devices are **never forwarded** to VLAN 20 ports, even though they share the same physical switch hardware.

## 4. VLAN IDs and the 802.1Q Standard

### 🎯 VLAN IDs

Every VLAN is identified by a **VLAN ID (VID)** — a number between **1 and 4094**.

| VLAN ID Range | Purpose |
|---|---|
| **1** | Default VLAN — all ports belong here unless reassigned |
| **2 – 1001** | Standard range available for general use |
| **1002 – 1005** | Reserved for legacy protocols (Token Ring, FDDI) |
| **1006 – 4094** | Extended range, supported on most modern switches |

> 🔑 VLAN 1 exists by default on nearly all switches and carries many control protocols by default. Industry best practice is to avoid using VLAN 1 for user, management, or sensitive traffic.

### 🎯 IEEE 802.1Q

**IEEE 802.1Q** is the industry standard defining how VLANs are implemented and how VLAN information is carried between switches via **VLAN tagging** (Section 5).

```
 Standard Ethernet Frame:
 [ Destination MAC | Source MAC | Type | Data ]

 802.1Q Tagged Frame:
 [ Destination MAC | Source MAC | 802.1Q Tag | Type | Data ]
                                      |
                              [ Tag Protocol ID | Priority | VLAN ID (12 bits) ]
```

The **12-bit VLAN ID field** is why VLAN IDs are limited to the range 1–4094 (`2^12 = 4096` values, minus 0 and 4095 which are reserved).

## 5. VLAN Tagging and Trunking

### 🎯 The Problem: Multiple VLANs Across Multiple Switches

When traffic from multiple VLANs needs to travel between two switches (or between a switch and a router), sending each VLAN over its own dedicated cable would require a massive number of physical links. Instead, **trunking** allows all VLANs to share a single physical link.

```
 [ Switch A ] ======= Trunk Link (one cable, all VLANs) ======= [ Switch B ]
               carries VLAN 10, VLAN 20, VLAN 30 simultaneously
```

### 🎯 How VLAN Tagging Works

When a frame from VLAN 10 needs to travel across a trunk link, the sending switch **inserts an 802.1Q tag** into the frame containing the VLAN ID. The receiving switch reads this tag to know which VLAN the frame belongs to, and delivers it to the correct VLAN on its side.

```
 PC-A (VLAN 10) sends a frame
         |
 Switch A adds 802.1Q tag: VLAN ID = 10
         |
 Frame travels across trunk link
         |
 Switch B reads tag: VLAN 10
         |
 Switch B forwards to all VLAN 10 ports (tag removed before delivery)
```

> 🔑 VLAN tags are added and removed by switches — **end devices never see the tag**. A PC doesn't know or care which VLAN it's in; the switch handles all tagging transparently.

## 6. Access Ports vs. Trunk Ports — In Depth

Recall from Chapter 17 (Section 12) that switch ports have two fundamental types. Now with VLAN context, the distinction becomes clearer:

| Feature | Access Port | Trunk Port |
|---|---|---|
| **Carries** | One VLAN only | Multiple VLANs simultaneously |
| **Tagging** | Frames are **untagged** — no 802.1Q tag | Frames are **tagged** with 802.1Q VLAN IDs |
| **Connected To** | End devices (PCs, phones, printers) | Other switches, routers, or Layer 3 devices |
| **Device Awareness** | End device is unaware of VLANs | The connected device must understand 802.1Q tags |

```
 [ PC ] ---Access Port (VLAN 10, untagged)--- [ Switch ] ---Trunk Port (tagged)--- [ Switch ]
```

## 7. The Native VLAN

### 🎯 What It Is

The **native VLAN** is a special VLAN designation on a trunk port — frames belonging to the native VLAN are sent **untagged** across the trunk link, while all other VLANs are tagged.

### 🧠 Why It Exists

802.1Q trunk ports must handle frames that arrive **without a tag** (e.g., from a device that doesn't support VLANs) — the native VLAN determines which VLAN these untagged frames are assigned to.

> 🔑 **Security note:** Mismatched native VLANs between two connected switches can cause frames to "jump" from one VLAN to another — a misconfiguration that is directly exploited by **VLAN hopping attacks** (Section 12). Industry best practice is to change the native VLAN from the default (VLAN 1) to an unused VLAN ID.

## 8. Inter-VLAN Routing

VLANs are designed to be isolated from each other — but sometimes controlled communication **between** VLANs is necessary (e.g., a Sales laptop accessing a file server on the Servers VLAN). This requires **inter-VLAN routing**.

### 🎯 Option 1: Router-on-a-Stick

A single physical link connects the switch to a router, configured as a **trunk port** carrying multiple VLANs. The router has **sub-interfaces** — one logical interface per VLAN — and routes traffic between them.

```
 [ Router ]
      | (trunk link with sub-interfaces per VLAN)
 [ Switch ]
  Port 1 (VLAN 10) -- PC-A (Sales)
  Port 2 (VLAN 20) -- PC-B (Engineering)

 PC-A communicates with PC-B -> traffic goes up to the router, routed between VLANs,
 and comes back down to the switch
```

### 🎯 Option 2: Layer 3 Switch

A **Layer 3 switch** (*Chapter 17, Section 11*) performs inter-VLAN routing internally — faster than router-on-a-stick, and the most common solution in modern enterprise networks.

```
 [ Layer 3 Switch ]
  VLAN 10 SVI (192.168.10.1) -- gateway for VLAN 10 devices
  VLAN 20 SVI (192.168.20.1) -- gateway for VLAN 20 devices
  Routing happens internally at hardware speed
```

> 🔑 A **SVI (Switch Virtual Interface)** is a logical interface on a Layer 3 switch representing a VLAN — it acts as the default gateway for all devices in that VLAN.

### Default Gateway Relationship

Every VLAN typically has its own default gateway.

Example:

- VLAN 10 → 192.168.10.1
- VLAN 20 → 192.168.20.1
- VLAN 30 → 192.168.30.1

Devices send traffic destined for other VLANs to their VLAN's default gateway, which performs the routing decision.

## 9. Common VLAN Types in Practice

| VLAN Type | Purpose | Example |
|---|---|---|
| **Data VLAN** | Carries standard user traffic | Employee workstations, laptops |
| **Voice VLAN** | Carries VoIP phone traffic, often with Quality of Service (QoS) priority | IP desk phones |
| **Management VLAN** | Used for administrative access to switches and network devices | SSH/web management of switches |
| **Native VLAN** | Handles untagged traffic on trunk ports (Section 7) | Typically reassigned from default VLAN 1 |
| **Guest VLAN** | Provides limited Internet-only access to untrusted devices | Guest Wi-Fi, contractor devices |
| **IoT VLAN** | Isolates Internet of Things devices from the main network | Smart TVs, cameras, sensors |

## 10. VLANs and Subnets — The Relationship

VLANs and subnets (*Chapter 6*) are closely related but operate at different layers:

| Concept | Layer | Purpose |
|---|---|---|
| **VLAN** | Data Link Layer (Layer 2) | Creates logical broadcast domain isolation at the switching layer |
| **Subnet** | Network Layer (Layer 3) | Defines a logical IP address range |

### 🧠 How They Work Together

In practice, VLANs and subnets are almost always configured as a **1-to-1 pair**:

```
 VLAN 10 (Sales)      <-> 192.168.10.0/24
 VLAN 20 (Engineering) <-> 192.168.20.0/24
 VLAN 30 (Guests)      <-> 192.168.30.0/24
```

Each VLAN gets its own subnet, and traffic between VLANs/subnets is controlled through inter-VLAN routing (Section 8) and firewall rules.

> 🔑 **Key insight:** A VLAN provides **Layer 2 isolation**; the corresponding subnet provides **Layer 3 addressing**. Together, they create a fully isolated network segment that is controlled and secure at both layers.

## 11. How This Works in Real Life

| Environment | VLANs in Practice |
|---|---|
| **Home Networks** | Consumer home routers rarely implement VLANs; some higher-end home routers support a separate Guest VLAN for visitor Wi-Fi |
| **Enterprise Networks** | VLANs are foundational — almost every enterprise network uses them extensively to separate departments, devices, and security zones |
| **Data Centers** | VLANs segment server roles (web tier, database tier, management) and are often combined with more advanced virtualization technologies |
| **Cloud Environments** | Cloud providers use VLAN-equivalent mechanisms (VPC subnets, security groups) to achieve the same logical isolation in virtualized infrastructure |
| **SOC** | Analysts use VLAN boundaries as reference points when investigating lateral movement — traffic crossing VLAN boundaries unexpectedly is a significant indicator of compromise |

## 12. VLANs and Cybersecurity

| Security Context | Relevance |
|---|---|
| **Network Segmentation** | VLANs are the primary technical mechanism implementing network segmentation (*Chapter 34*) — isolating sensitive systems, reducing lateral movement risk |
| **VLAN Hopping — Switch Spoofing** | An attacker configures their device to negotiate a trunk link with the switch, gaining access to all VLANs on that trunk — defended by disabling dynamic trunking on user-facing ports |
| **VLAN Hopping — Double Tagging** | An attacker sends a frame with two 802.1Q tags; the first switch strips the outer tag (native VLAN), and the second switch delivers the frame to the VLAN specified in the inner tag — escaped into an otherwise inaccessible VLAN |
| **Native VLAN Mismatches** | Misconfigured native VLANs can cause unintended traffic crossing between VLANs — always change the native VLAN from VLAN 1 to an unused ID |
| **Isolating Untrusted Devices** | Guest Wi-Fi, IoT devices, and contractor machines should always be placed on dedicated VLANs, separated from sensitive internal systems |
| **VLAN as a Defense Layer** | Even if an attacker compromises one device on a VLAN, VLAN segmentation limits how far they can move without crossing a controlled inter-VLAN routing boundary |

> 🔑 **VLAN hopping** is not a theoretical concern — it's a documented, demonstrable attack. The two primary defenses are: never use VLAN 1 as the native VLAN, and explicitly configure all non-trunk ports as **access ports** with dynamic trunking disabled.

## 13. Troubleshooting Perspective

| Problem | Symptom | What Engineers Check First | Common Fix |
|---|---|---|---|
| Device can't reach others on the same logical group | Connectivity fails within what should be one group | Whether the port is assigned to the correct VLAN | Reassign the port to the correct VLAN |
| Device can't reach its default gateway | No Internet/inter-VLAN access despite correct IP | Whether the gateway IP corresponds to the correct VLAN's SVI | Verify SVI/router sub-interface configuration |
| VLANs not passing between switches | Traffic works locally but not between switches | Whether the inter-switch link is configured as a trunk, and whether the VLAN is in the trunk's allowed list | Configure trunk port, add VLAN to allowed list |
| Unexpected traffic between VLANs | Devices on separate VLANs can reach each other when they shouldn't | Native VLAN mismatch, VLAN hopping misconfiguration | Correct native VLAN, disable dynamic trunking on access ports |

## 14. Real-World Walkthrough: Designing a Segmented Office Network

**Scenario:** A small company has one physical switch and needs separate networks for Staff, Guests, Servers, and IP Phones.

**Step 1 — Plan VLANs and corresponding subnets:**

| VLAN | Name | Subnet | Purpose |
|---|---|---|---|
| VLAN 10 | Staff | 192.168.10.0/24 | Employee workstations |
| VLAN 20 | Servers | 192.168.20.0/24 | Internal file/application servers |
| VLAN 30 | Voice | 192.168.30.0/24 | VoIP phones |
| VLAN 40 | Guest | 192.168.40.0/24 | Visitor Wi-Fi, Internet-only |

**Step 2 — Configure switch ports:**
- Ports 1–10: Access ports → VLAN 10 (Staff workstations)
- Ports 11–14: Access ports → VLAN 20 (Servers)
- Ports 15–20: Access ports → VLAN 30 (Phones)
- Port 21: Access port → VLAN 40 (Guest Wi-Fi access point)
- Port 24: **Trunk port** → connected to Layer 3 switch/router for inter-VLAN routing

**Step 3 — Configure inter-VLAN routing:**
The Layer 3 switch or router handles routing between VLANs, with firewall rules allowing Staff to reach Servers, but preventing Guests from reaching anything internal.

**Step 4 — Security hardening:**
- Change native VLAN to an unused VLAN ID
- Disable dynamic trunking on all access ports
- Apply ACLs restricting what each VLAN can reach

## 15. Common Beginner Misconceptions

| ❌ Misconception | ✅ Reality |
|---|---|
| VLANs are only useful in large enterprise networks | Even small networks benefit from VLANs for guest isolation, IoT segmentation, and basic security |
| VLANs provide complete security isolation with no additional controls needed | VLANs provide Layer 2 isolation — but proper inter-VLAN firewall rules are also required for true security |
| A VLAN and a subnet are the same thing | A VLAN is a Layer 2 broadcast domain concept; a subnet is a Layer 3 addressing concept — they're different but almost always paired together |
| Devices on different VLANs can never communicate | They can, via inter-VLAN routing (*Chapter 16*) — VLANs isolate by default but communication can be explicitly permitted and controlled |
| VLANs require separate physical switches | VLANs exist specifically to provide logical separation on shared physical infrastructure |
| VLAN 1 should always be used as the management VLAN | VLAN 1 is the default VLAN that everything falls into if misconfigured — it's a security best practice to use a dedicated, non-default VLAN for management |
| VLAN hopping only works in theory | It's a real, demonstrable attack; proper port configuration is required to prevent it |
| All switch ports are trunk ports by default | All ports are **access** ports on VLAN 1 by default — trunk ports must be explicitly configured |
| Devices need to understand VLAN tags | End devices typically never see VLAN tags; switches add and remove tags automatically |

## 16. Common Terminology Glossary

| Term | Definition |
|---|---|
| **VLAN** | Virtual Local Area Network — a logical network segment within a physical switch |
| **VLAN ID (VID)** | A number (1–4094) identifying a specific VLAN |
| **IEEE 802.1Q** | The standard defining VLAN tagging for Ethernet frames |
| **VLAN Tag** | An 802.1Q header inserted into a frame identifying its VLAN |
| **Trunking** | Carrying multiple VLANs over a single physical link using tagging |
| **Access Port** | A port carrying untagged traffic for a single VLAN |
| **Trunk Port** | A port carrying tagged traffic for multiple VLANs |
| **Native VLAN** | The VLAN assigned to untagged frames on a trunk port |
| **Inter-VLAN Routing** | Routing traffic between different VLANs via a router or Layer 3 switch |
| **Router-on-a-Stick** | A routing topology using a single trunk link with sub-interfaces per VLAN |
| **SVI** | Switch Virtual Interface — a logical Layer 3 interface representing a VLAN on a Layer 3 switch |
| **Data VLAN** | A VLAN carrying standard user traffic |
| **Voice VLAN** | A VLAN carrying VoIP traffic, often with QoS priority |
| **Management VLAN** | A VLAN dedicated to administrative network device access |
| **Guest VLAN** | A VLAN providing limited, isolated access for untrusted devices |
| **IoT VLAN** | A VLAN isolating Internet of Things devices |
| **VLAN Hopping** | An attack gaining unauthorized access to a different VLAN |
| **Double Tagging** | A VLAN hopping technique using two nested 802.1Q tags |
| **Switch Spoofing** | A VLAN hopping technique where an attacker negotiates a trunk link |
| **Broadcast Domain** | The scope of devices receiving broadcast frames — one per VLAN |
| **Network Segmentation** | Dividing a network into isolated segments for security/performance (*Chapter 34*) |
| **Default VLAN** | The VLAN all switch ports belong to initially, typically VLAN 1 |
| **Allowed VLAN List** | The VLANs permitted to traverse a trunk link |
| **Tagged Frame** | Ethernet frame containing an 802.1Q VLAN tag |
| **Untagged Frame** | Ethernet frame without a VLAN tag |
| **Gateway VLAN Interface** | The Layer 3 interface providing routing services for a VLAN |

## 17. Common Interview Questions

1. **What is a VLAN and why is it used?** A logical subdivision of a physical network creating isolated broadcast domains — used for security, traffic reduction, organization, and efficiency.
2. **What is 802.1Q and what role does it play?** The IEEE standard defining how VLAN tags are added to Ethernet frames to carry VLAN information across trunk links.
3. **What's the difference between an access port and a trunk port?** An access port carries untagged traffic for one VLAN; a trunk port carries tagged traffic for multiple VLANs.
4. **What is the native VLAN?** The VLAN that handles untagged frames arriving on a trunk port — a security best practice is to change it from the default VLAN 1.
5. **How does inter-VLAN routing work?** Traffic between VLANs must pass through a router or Layer 3 switch, which can apply access control rules at the routing boundary.
6. **What is VLAN hopping and how is it prevented?** An attack gaining unauthorized VLAN access — prevented by explicitly configuring access ports (disabling dynamic trunking) and changing the native VLAN from VLAN 1.
7. **What is double tagging?** A VLAN hopping technique using two 802.1Q tags — the outer tag matches the native VLAN, getting stripped by the first switch, while the inner tag directs the frame to the target VLAN on subsequent switches.
8. **How do VLANs and subnets relate?** They typically pair 1-to-1 — the VLAN provides Layer 2 isolation, the subnet provides Layer 3 addressing for that segment.
9. **What is a Switch Virtual Interface (SVI)?** A logical Layer 3 interface on a Layer 3 switch representing a VLAN, acting as the default gateway for devices in that VLAN.
10. **Why should VLAN 1 not be used for sensitive traffic?** It's the default VLAN — any misconfigured or unassigned port falls into VLAN 1, making it a poor choice for sensitive or management traffic.

## 18. Key Takeaways

### 📋 Summary

- A **VLAN** logically divides a physical switch into multiple isolated broadcast domains, providing security, organization, and efficiency without requiring separate physical hardware.
- VLANs are identified by **VLAN IDs (1–4094)** and standardized by **IEEE 802.1Q**.
- **Trunk ports** use 802.1Q tagging to carry multiple VLANs across a single link; **access ports** carry untagged traffic for a single VLAN.
- The **native VLAN** handles untagged traffic on trunk ports — always change it from the default VLAN 1.
- VLANs are isolated by default; **inter-VLAN routing** via a router or Layer 3 switch enables controlled communication between them.
- VLANs and subnets pair **1-to-1** — VLAN for Layer 2 isolation, subnet for Layer 3 addressing.
- **VLAN hopping** (switch spoofing and double tagging) are real attacks prevented through proper port configuration and native VLAN hardening.
